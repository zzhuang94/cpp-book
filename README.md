# C++11 工程上手手册

这套文档写给 **至少掌握一门其它编程语言**、想系统进入 `C++11` 工程实践的读者：**不要求工作年限，也不要求已经是「资深」后端或客户端工程师**。

它的目标不是把 `C++` 全部标准细节铺满，而是围绕 `C++11` 建立一套足够稳定的工程直觉：让你在读完之后，能够进入基于 `C++11` 编译的 **团队项目或有一定体量的代码库**，理解它的代码组织、资源模型、构建方式和常见工程写法，并开始做真实修改。

----

# 这套文档适合谁

如果你符合下面这些情况，这套文档基本就是为你准备的：

- 你主要写 `Go`、`PHP`、`Java`、`C#`、`JS/TS` 或其它语言，**在那一门语言里写过完整小程序**（分支、循环、函数、模块化即可）。
- 你对 `C/C++` 工程缺少稳定手感，或只在课本里接触过语法。
- 你大致知道指针、引用、类、模板这些词，却很难把它们串成工程能力。
- 你准备参与或接手一个 `C++11` 项目，希望先补齐最关键的一圈知识。
- 你不想先啃标准条文，而是更关心“真实项目里怎么读、怎么改、怎么排错”。

若你对指针、引用、类只有模糊印象，建议先通读「基础教程」，再重点读 [指针](advanced/pointer.md)、[引用](advanced/ref.md)，然后进入「现代 C++11 核心」。与常见语言的差异可先扫一眼 [语言对照](start/compare.md)。

**本文不面向零编程基础读者**：若还不会分支、循环、函数，请先完成任意一门语言的入门课，再回来读本文。

# 为什么以 C++11 为主

这套文档会明确以 `C++11` 为主线。

原因很简单：对很多真实项目来说，`C++11` 是现代 `C++` 的起点，也是今天仍然大量在线上系统、基础库、客户端程序和服务端组件中使用的版本。

很多关键能力都从这里稳定下来：

- `auto`
- `nullptr`
- 范围 `for`
- `enum class`
- `lambda`
- 右值引用与移动语义
- `unique_ptr` / `shared_ptr`
- `override` / `final`
- `=default` / `=delete`
- `chrono`
- 线程库
- `unordered_map`

如果 `C++11` 这一层没有站稳，继续看 `C++14/17/20` 往往只会更乱。  
所以主线会始终围绕 `C++11` 展开；如果确实有必要提到更高版本的常见能力，会单独放进补充章节，帮助你识别项目里的新写法，但不会让主线偏掉。

# 这套文档真正想解决什么

对很多其它语言开发者来说，进入 `C++` 最难的通常不是某个单点语法，而是下面这些整体问题：

- 为什么 `C++` 项目里要关心头文件、声明、定义、编译、链接。
- 为什么对象离开作用域自动析构，会直接影响资源管理设计。
- 为什么“传值 / 传引用 / 传指针”不是表面语法，而是接口语义。
- 为什么现代 `C++` 不鼓励散落的 `new/delete`。
- 为什么容器、算法、迭代器和 RAII 会反复出现在项目代码里。
- 为什么很多历史代码还能编译，但并不是今天推荐的默认写法。
- 为什么在团队项目或维护期较长的代码库里，测试、调试、Sanitizer、构建系统和第三方依赖同样属于基础能力。

这套文档会把这些问题拆开讲，并尽量用短小、完整、可编译的例子，帮助你把抽象概念和真实项目代码对应起来。

# 文档主线

目前整套文档按下面几条主线组织：

## 1. 开始

先建立整体地图，明确 `C++11` 的重点、构建模型和学习顺序。

## 2. 基础教程

这一部分的内容是有意拆细的，目的是让读者在第一次系统学习后，后续也能把它当成查阅入口。

阅读策略可以二选一（不必纠结，中途也可以换轨）：

- **路径 A（偏赶项目、已有一定抽象习惯）**：不要平均用力；按下面「分组」跳读优先子集（见后文列表），再进现代核心。
- **路径 B（基础更弱、希望稳扎稳打）**：**按侧边栏顺序通读「基础教程」全文**，`修饰符`、`时间/日期` 仍可放在第一轮之后补读，再进入「现代 C++11 核心」。

若走路径 A，更适合按下面几组来读：

- 语法入口：`基础语法`
- 类型与声明：`基础类型`、`变量`、`常量`
- 表达式与控制流：`修饰符`、`运算符`、`条件`、`循环`、`函数`
- 常用数据与输入输出：`数组`、`字符串`、`时间/日期`、`输入/输出`

如果只从“尽快参与项目”的角度看，这一部分里最值得优先读透的是：

- `基础语法`
- `基础类型`
- `变量`
- `常量`
- `运算符`
- `条件`
- `循环`
- `函数`
- `字符串`
- `输入/输出`

相对来说，`修饰符` 和 `时间/日期` 更适合放在第一轮之后补读。  
也就是说，基础教程不是内容不够，而是作为目录已经足够丰富；更合适的优化方式不是继续拆细，而是在阅读顺序上避免平均用力。

## 3. 现代 C++11 核心

这是最关键的一段，路径 A 下不必按侧边栏顺序机械读完。  
如果目标是参与 **有一定规模或多人协作的** `C++11` 代码库，更推荐按下面四层理解：

第一层：资源模型

- `指针`
- `引用`
- `所有权与 RAII`

这是最优先的一层，因为这类项目里最容易卡住人的，通常不是语法，而是所有权、借用关系、生命周期和移动语义。

第二层：标准库与错误边界

- `容器/算法/迭代器`
- `异常与错误处理`

这层决定你是否能把常见业务代码真正读顺。

第三层：抽象与泛型

- `函数进阶`
- `函数重载`
- `运算符重载`
- `模板`

这一层更偏“读懂框架代码、基础设施代码和泛型代码”。  
其中 `函数进阶` 实际上承担了 `lambda`、函数对象、`std::function`、右值引用参数等高频主题；对真实工程来说，它的重要性比标题看起来更高。

第四层：并发与场景化主题

- `多线程`
- `网络编程基础`
- `高并发`

这层里的知识点在体量较大或服务型项目里当然常见，但它们更依赖项目类型。  
如果你参与的是通用业务系统、工具链、客户端、基础库或离线程序，那么 `多线程` 值得优先看，而 `网络编程基础` 和 `高并发` 可以放到后面按需补。

从「能读懂并改真实代码」这个目标看，这一部分已经覆盖了最关键的一圈知识点：

- 指针、引用、所有权、RAII、智能指针、移动语义
- `lambda`、函数对象、`std::function`
- 容器、算法、迭代器
- 模板与泛型基础
- 异常与错误边界
- 线程与同步原语

真正不该平均看待的，不是“有没有缺主题”，而是“哪些必须第一轮吃透，哪些可以按项目类型后读”。

## 4. 面向对象

进入类设计、继承、多态，以及现代 `C++11` 对象语义。

## 5. 工程实践

把语言能力真正接到项目里，包括：

- 代码组织
- 编译与链接
- 构建工具与 `CMake`
- 测试与验证
- 调试与诊断工具链
- 第三方依赖
- 接手现有项目的方法

## 6. 安全与风格

把未定义行为、安全边界、调试习惯和现代 `C++11` 默认写法单独拉出来，避免它们被当成“随便看看”的补充材料。

# 读完之后应该获得什么能力

这套文档的目标不是让你立刻成为长期专精的 `C++` 专家，而是尽快获得下面这些能力：

- 能独立编译和运行最小 `C++11` 程序。
- 能理解头文件、源文件、翻译单元、编译、链接之间的关系。
- 能读懂值语义、引用语义、对象生命周期和资源所有权。
- 能识别什么时候该用容器、算法、智能指针和 RAII。
- 能看懂常见 `C++11` 类、多态接口和标准库代码风格。
- 能跑测试、读调试信息、使用基本诊断工具定位常见问题。
- 能从构建入口、目录结构、公共接口和测试代码开始接手一个真实项目。
- 能在项目中做小到中等规模修改，而不是只敢看不敢动。

# 推荐阅读顺序

下面默认 **路径 A**（想尽快进项目）；若你更适应 **路径 B**，把第 3 步改成「按侧边栏顺序通读基础教程」，其余阶段顺序可保持不变。

如果你的目标是“尽快参与项目”，推荐按下面顺序：

1. 先看 `简介`、`前言`、`语言对照`、`C++11 核心地图`。
2. 再看 `Hello World` 和 `头文件与构建模型`，打通最小编译链路。
3. 再按“基础教程”的分组来读（路径 A），而不是机械地逐篇顺序读完：
   - 先读 `基础语法`、`基础类型`、`变量`、`常量`
   - 再读 `运算符`、`条件`、`循环`、`函数`
   - 再读 `字符串`、`数组`、`输入/输出`
   - `修饰符`、`时间/日期` 放在第一轮后补
4. 接着进入“现代 C++11 核心”的第一层和第二层：
   - `指针`
   - `引用`
   - `所有权与 RAII`
   - `容器/算法/迭代器`
   - `异常与错误处理`
5. 然后补“现代 C++11 核心”的抽象层：
   - `函数进阶`
   - `模板`
   - 再按需阅读 `函数重载`、`运算符重载`
6. 接着进入 `类和对象`、`继承`、`多态`，把对象语义和接口设计真正接起来。
7. 再看 `调试入门`、`安全边界与未定义行为`、`测试与验证`、`诊断工具链`。
8. 最后进入 `代码组织与项目阅读`、`编译与链接`、`CMake 实战`、`第三方库与依赖管理`、`项目接手`。

如果你的项目明显是网络服务或高并发系统，再把：

- `多线程`
- `网络编程基础`
- `高并发`

提前到第 5 步或第 6 步之间。

如果你已经手上有真实项目，也可以采用更实用的节奏：

1. 先读一章。
2. 再去项目里找两到三段对应代码。
3. 再回来看文档，把“概念”和“实际写法”对应起来。

这种方式通常比纯看教程更容易形成手感。

# 如何使用这套文档

建议不要只看，要交替做下面几件事：

1. 阅读一章，先理解概念和推荐写法。
2. 手敲示例代码并编译运行。
3. 故意制造一些错误，观察编译器、链接器和运行结果。
4. 回头在真实项目里找相似写法。
5. 记录“这个写法在工程里通常意味着什么”。

`C++` 很多知识如果只停留在阅读层面，会一直处于“似懂非懂”的状态；只有和编译、报错、调试、项目代码绑定起来，理解才会真正稳定。

----

# 关于旧代码和更高版本代码

真实项目里你很可能同时遇到三种东西：

- 老式 `C` / `C++98` 风格代码
- 已经现代化到 `C++11` 的代码
- 掺杂了一些 `C++14/17/20` 新特性的代码

本文的主线永远以 `C++11` 为核心。

这意味着：

- 主要讲解会优先落在 `C++11` 就能稳定使用的能力上。
- 对旧代码，会解释你必须知道的背景和风险。
- 对更高版本，会单独说明“这是什么、为什么项目里会出现、和 `C++11` 相比多了什么”，但不会让主体章节偏离 `C++11`。

# 这套文档不打算做什么

为了保证节奏，这套文档会刻意避免几件事：

- 不追求逐条覆盖全部标准细节。
- 不把冷门角落特性都讲成独立专题。
- 不把历史写法一刀切地全部说成错误。
- 不把“会写某段示例代码”等同于“已经具备工程能力”。

它更像一套“进入 `C++11` 工程的上手手册”，而不是一本标准参考书。
