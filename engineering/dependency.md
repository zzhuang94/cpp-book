# 第三方库与依赖管理

> 写练习代码时，很多人只需要标准库；  
> 但一进入真实工程，几乎一定会遇到第三方库：日志库、测试框架、网络库、格式化库、序列化库、数据库驱动……

这时候问题就不再是“这个库好不好用”，而是：

- 怎么把它接进项目
- 怎么保证团队里每个人都能稳定构建
- 怎么控制版本
- 怎么避免依赖越来越乱
- `find_package()`、`vcpkg`、`conan` 到底分别在解决什么问题

这篇文档把读者默认当作刚接触工程实践的初学者来介绍。你会看到：

- 什么是第三方依赖，为什么依赖管理会变成一个工程问题
- 头文件库、静态库、动态库在接入上有什么差别
- `CMake` 里如何使用 `find_package()`
- `vcpkg` 和 `conan` 各自是什么、适合什么场景
- 项目里应该如何选择依赖管理策略
- 日常开发中和依赖相关的常见坑

----

# 先建立整体认知

## 什么是依赖

依赖（dependency）可以简单理解成：

**你的项目不是独立完成所有功能，而是需要借助别人已经写好的代码。**

例如：

- 你想更方便地格式化字符串，可能会用 `fmt`
- 你想写单元测试，可能会用 `googletest`
- 你想做日志，可能会用 `spdlog`
- 你想处理 JSON，可能会用 `nlohmann/json`

这些外部代码，都是你的项目依赖。

## 为什么依赖管理会成为问题

如果只是自己电脑上随手写个 demo，依赖管理似乎很简单：

- 下载源码
- 扔进项目
- 能编过就行

但真实项目很快会遇到这些问题：

- 团队成员机器环境不一致
- 某个库升级后，旧代码不兼容
- CI 上构建失败，但本地能过
- Debug 和 Release 用了不同版本的库
- 不同平台安装方式完全不一样
- 某些依赖只在实现层使用，某些依赖却泄漏进了公共接口

也就是说，依赖管理的核心不是“把库接进来一次”，而是：

**让依赖能被长期、稳定、可重复地使用。**

# 第三方库的几种常见形态

## 1. 头文件库（Header-only）

这种库只有头文件，没有单独的二进制库文件。  
使用方式通常很简单：

```cpp
#include <nlohmann/json.hpp>
```

这类库的特点是：

- 接入简单
- 不需要单独链接
- 很适合模板库和小型通用工具库
- 但头文件可能很重，编译时间会增加

## 2. 静态库

静态库会在链接阶段被打进你的最终程序。

常见扩展名：

- Windows: `.lib`
- Linux/macOS: `.a`

特点：

- 部署相对简单
- 运行时通常不需要额外找库
- 但最终程序体积可能更大

## 3. 动态库

动态库在运行时由系统加载。

常见扩展名：

- Windows: `.dll`
- Linux: `.so`
- macOS: `.dylib`

特点：

- 多个程序可以共享同一份库
- 升级库时更灵活
- 但部署更复杂
- 很容易出现“编译没问题，运行时报缺库”

## 为什么要先分清形态

因为不同形态带来的接入步骤不同：

- 头文件库重点是头文件路径
- 静态库重点是头文件路径 + 链接
- 动态库重点是头文件路径 + 链接 + 运行时可用性

很多初学者一开始接第三方库，总以为 “`#include` 成功了就算接好了”，其实远远不够。

# CMake 里如何使用第三方库

> 在现代 `CMake` 项目里，接入第三方库最重要的思想不是“拼路径”，而是“使用 target”。

## 最理想的情况：库提供了 CMake 包配置

很多现代库都支持：

```cmake
find_package(fmt REQUIRED)
target_link_libraries(app PRIVATE fmt::fmt)
```

这几行已经表达了完整依赖关系：

- 找到 `fmt`
- 如果找不到就报错
- 找到后把它链接给 `app`

这时你通常不需要自己再手工写：

- 头文件目录
- 库目录
- 裸库文件名

因为这些信息已经封装在 `fmt::fmt` 这个导入目标里了。

## `find_package()` 是什么

`find_package()` 可以理解成：

**让 `CMake` 去查找并加载某个依赖的构建信息。**

例如：

```cmake
find_package(spdlog REQUIRED)
target_link_libraries(app PRIVATE spdlog::spdlog)
```

如果这个库安装得正确，并且提供了标准 `CMake` 配置，那么 `CMake` 就能知道：

- 头文件在哪
- 需要链接哪些库
- 可能还需要什么编译定义

## 为什么导入目标比手工写路径更好

假设你手工写：

```cmake
include_directories(/some/path/include)
link_directories(/some/path/lib)
target_link_libraries(app PRIVATE fmt)
```

这种写法的问题包括：

- 可移植性差
- 路径容易写死
- 哪些配置属于哪个依赖，不清晰
- 依赖传播关系不明确

而导入目标会更像：

- 一个“打包好的依赖对象”
- 包含头文件路径、链接信息、编译属性

这就是现代 `CMake` 推崇 target 风格的原因之一。

# 没有 find_package 的时候怎么办

并不是所有库都提供完善的 `CMake` 配置。  
这时常见做法有几种。

## 方式 1：把依赖源码放进项目

例如：

```text
third_party/
  somelib/
```

如果这个库自己就支持 `CMake`，你可能可以：

```cmake
add_subdirectory(third_party/somelib)
target_link_libraries(app PRIVATE somelib)
```

优点：

- 可控
- 容易跟项目一起构建
- 新机器拉代码后更容易直接跑起来

缺点：

- 仓库会变大
- 升级依赖更麻烦
- 项目里会混入不少第三方源码

## 方式 2：手工指定路径

这是一种更原始但仍然常见的做法。  
例如：

```cmake
target_include_directories(app PRIVATE /path/to/lib/include)
target_link_libraries(app PRIVATE /path/to/lib/libsomelib.a)
```

它能工作，但通常不是优先方案。  
因为：

- 可移植性差
- 不利于团队协作
- 很容易把本机路径写死

## 方式 3：自己写 Find 模块或包配置

这已经是进阶主题了。  
有些项目会自己写：

- `FindXXX.cmake`
- 或导出自己的包配置文件

这样可以让依赖接入更标准化。  
对初学者来说，先知道有这条路即可，不需要一上来就掌握。

# vcpkg 是什么

> `vcpkg` 可以理解成一个偏向 `C/C++` 生态的包管理工具，尤其在 Windows 和 `CMake` 生态里很常见。

## 它在解决什么问题

很多第三方库接入困难，不是因为库本身难用，而是因为：

- 安装方式不统一
- 不同平台操作不同
- 库之间还有依赖链
- 本地环境和 CI 环境很难保持一致

`vcpkg` 的目标之一，就是帮助你：

- 下载依赖
- 构建依赖
- 安装依赖
- 让 `CMake` 更容易找到这些依赖

## 一个很粗略的理解

你可以把 `vcpkg` 想成：

- 针对 `C/C++` 库的包管理器
- 同时和 `CMake` 集成得比较自然

## 常见使用方式

例如，你可能先安装库：

```bash
vcpkg install fmt
vcpkg install spdlog
```

然后在 `CMake` 中：

```cmake
find_package(fmt REQUIRED)
find_package(spdlog REQUIRED)

target_link_libraries(app PRIVATE
    fmt::fmt
    spdlog::spdlog
)
```

如果 `vcpkg` 和 `CMake` 集成得正确，`find_package()` 就能找到这些包。

## 它的优点

- 和 `CMake` 生态配合较好
- Windows 下体验通常不错
- 安装第三方库比手工配置轻松很多
- 对很多常见库支持成熟

## 它的注意点

- 团队里要统一依赖版本
- 不同 triplet、不同构建类型可能带来差异
- “本地能用”不代表 CI 就自动一致

## manifest 模式

现代 `vcpkg` 里很重要的一种方式是 manifest 模式，也就是项目里放一个：

```text
vcpkg.json
```

由它描述项目依赖。  
这样依赖信息就变成了项目源码的一部分，而不是“每个人自己装了什么”。

这是工程上非常有价值的一步。

----

# conan 是什么

> `conan` 也是 `C/C++` 领域非常常见的包管理器，但它更强调包、版本、配置和二进制复用。

## 它在解决什么问题

`conan` 主要想解决：

- 依赖版本管理
- 编译配置差异管理
- 二进制包复用
- 团队和 CI 的可重复构建

对很多大型工程来说，它的吸引力在于：

- 依赖关系表达力更强
- 版本和配置管理更系统化
- 很适合复杂依赖图

## 粗略理解

如果说 `vcpkg` 更像“安装库并让 `CMake` 找到它们”，  
那么 `conan` 更像“围绕包、版本、配置和构建结果做更完整的依赖管理”。

这个说法不完全精确，但对初学者足够有帮助。

## 常见使用方式

一个项目里可能会有：

```text
conanfile.txt
```

或者：

```text
conanfile.py
```

用来描述依赖。  
然后你通过 `conan install` 拉取和生成构建配置，再让 `CMake` 使用这些信息。

## 它的优点

- 对复杂依赖管理能力很强
- 更适合强调版本控制和复现性的工程
- CI 场景下很有价值
- 可以管理很多编译配置组合

## 它的注意点

- 理解成本相对更高
- 概念比“直接装库”要更多
- 对零基础读者来说，一开始会比 `vcpkg` 更抽象一些

----

# vcpkg 和 conan 怎么选

> 没有绝对标准答案，关键看项目规模、团队环境和你想解决的问题。

可以先用一个非常粗略的判断：

## 更偏向 vcpkg 的情况

- 你主要在 `CMake` 生态里工作
- 想尽快把常见第三方库接进来
- 希望接入方式简单直接
- 工程复杂度还没那么高
- 尤其是在 Windows 环境下

## 更偏向 conan 的情况

- 你更在意依赖版本和构建配置的系统管理
- 项目依赖图更复杂
- 团队和 CI 需要更强的可重复性
- 你愿意接受更高一些的学习成本

## 对初学者的现实建议

如果你只是刚进入工程实践阶段，可以先这样走：

1. 先学会 `find_package()` 和 target 风格链接
2. 再接触 `vcpkg`
3. 等你对工程依赖有更多真实痛点，再进一步理解 `conan`

这样会更自然。

----

# 依赖管理策略怎么选

## 策略 1：系统安装

直接靠系统包管理器或手工安装库。

优点：

- 简单
- 本地试验快

缺点：

- 团队环境难统一
- CI 不够稳定
- 机器迁移成本高

## 策略 2：项目内 vendor / third_party

把依赖源码放进仓库。

优点：

- 可复现性强
- 新环境更容易拉起

缺点：

- 仓库容易变大
- 升级和维护更麻烦

## 策略 3：包管理器

使用 `vcpkg`、`conan` 等工具。

优点：

- 更适合长期维护
- 更容易做版本固定
- 团队协作更顺畅

缺点：

- 需要额外学习和约束
- 项目会多一层工具依赖

## 哪种最好

不是哪种“技术上最先进”就一定最好，而是哪种最适合当前项目。

对很多团队来说，一个实用的目标是：

**依赖来源清楚、版本可控、构建可复现。**

只要能稳定达到这三点，方案通常就算合格。

# 依赖在代码层面的注意点

> 依赖管理不只是构建工具问题，也会反过来影响代码组织。

## 1. 不要让第三方类型无节制泄漏进公共接口

例如，假设你公共头文件里直接到处暴露某个第三方日志库的具体类型：

- 以后想替换库会非常痛苦
- 所有使用者都会被这个依赖绑定
- 编译依赖会被放大

更好的方式通常是：

- 模块内部使用第三方库
- 对外暴露自己的抽象接口

## 2. 区分“实现依赖”和“接口依赖”

这是工程里非常重要的一层思维。

### 实现依赖

某个库只是你 `.cpp` 里内部实现需要用到。  
这时它最好留在实现层，不要进入公共头文件。

### 接口依赖

某个库的类型已经出现在你的公共头文件里。  
那它就不再只是实现依赖，而变成了接口的一部分。

一旦走到这一步，你的使用者也必须理解并接受这个依赖。

## 3. 依赖越底层，升级要越谨慎

如果一个依赖处在很基础的位置，例如：

- 核心工具库
- 基础网络库
- 公共日志库

那么它一旦升级，影响面就会非常大。  
这类依赖要比边缘工具库更谨慎地升级。

## 4. 固定版本比“永远最新版”更重要

工程稳定性通常来自：

- 明确版本
- 可复现构建

而不是“每次都拉最新版本看看能不能过”。

对学习阶段来说，用最新版体验新特性没问题；  
对项目维护来说，可控往往比“最新”更重要。

# 常见坑

## 1. 本地能编，别人机器不行

高频原因包括：

- 你本地装了某些库，但项目没记录
- 路径写死在本机目录里
- 没有固定依赖版本

## 2. include 能找到，链接却失败

这通常意味着：

- 头文件路径解决了
- 但库没有正确链接

也就是：

- 编译通过
- 链接失败

## 3. 链接通过，运行时报缺库

这通常是动态库问题。  
也就是说：

- 编译没问题
- 链接没问题
- 运行时系统找不到对应动态库

## 4. Debug 和 Release 混用

不同构建类型可能链接不同版本的库。  
如果混用，很容易出现诡异问题，尤其在 Windows 工具链里更要小心。

## 5. 第三方库升级后大面积出问题

这说明项目可能：

- 把第三方类型暴露得太深
- 缺少版本管理
- 缺少升级验证流程

----

# 一个实用的入门顺序

如果你刚开始接触依赖管理，可以按这个顺序来：

1. 先理解头文件库、静态库、动态库的区别
2. 学会在 `CMake` 里使用 `find_package()` 和 `target_link_libraries()`
3. 学会判断“编译问题、链接问题、运行时加载问题”
4. 再接触 `vcpkg`
5. 当项目规模更复杂时，再理解 `conan`

这个顺序的好处是：

- 先理解构建本质
- 再引入包管理器
- 不容易被工具细节反客为主

----

# 小结

第三方库与依赖管理，可以浓缩成一句话：

**真正困难的不是“把库接进来一次”，而是“让依赖在团队和项目生命周期里始终稳定、可控、可复现”。**

如果你只先记住最关键的几件事，可以抓住这些：

- 先分清头文件库、静态库、动态库
- 在 `CMake` 中优先使用 `find_package()` 和导入目标
- `vcpkg` 更适合从“安装和接入依赖”入门
- `conan` 更适合复杂项目中的版本和配置管理
- 依赖不只是构建问题，也会反过来影响代码接口设计

当你开始用“版本、可复现、依赖传播、接口边界”的视角去看第三方库时，你就已经在真正理解现代 `C++` 工程，而不只是“会把一个库 include 进来”。
