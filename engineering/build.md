# 编译与链接

> `C++11` 工程里，很多 *看起来像语法问题* 的报错，本质上其实是构建问题。  
> 你写的代码不是直接变成可执行程序的，而是要经过一整套 `预处理 -> 编译 -> 汇编 -> 链接` 流程。

很多人刚学 `C++` 时，关注点主要是：

- 语法是否正确
- 类怎么写
- 模板怎么用

但一进入真实项目，很快就会遇到另一类问题：

- 为什么头文件改了一行，几十个源文件都要重新编译
- 为什么代码能编过，却在链接时报 `undefined reference`
- 为什么用了第三方库之后，编译没问题，运行时却提示找不到 `.dll`
- 为什么 Debug 能过，Release 却出问题
- 为什么同一份代码在不同机器、不同编译器上行为不一致

这些问题背后，几乎都和 **编译与链接模型** 有关。

这篇文档会按“先建立整体认知，再进入细节”的顺序展开。你会看到：

- `C++11` 编译的四个阶段分别在做什么
- 翻译单元、目标文件、符号、链接器之间如何协作
- 静态库、动态库分别是什么，什么时候该用哪一种
- 工程开发中哪些习惯会直接影响构建稳定性和可维护性
- 如何快速区分“编译错误、链接错误、运行时加载错误”

----

# 整体模型

> 一个 `C++` 程序从源码到最终可执行文件，不是 *一次性变出来* 的，而是流水线式地逐步处理。

粗略来看，流程是这样的：

```text
源代码(.cpp/.h)
   ↓
预处理(preprocessing)
   ↓
编译(compilation)
   ↓
汇编(assembly)
   ↓
目标文件(.o/.obj)
   ↓
链接(linking)
   ↓
可执行文件 / 静态库 / 动态库
```

如果把这个过程类比成造房子：

- 头文件和源文件是原始材料与设计说明
- 预处理是在整理和展开材料
- 编译是在把高级语言翻译成更底层表示
- 汇编是在生成机器相关的目标文件
- 链接是在把多个零件装配成完整成品

这个模型直接决定了很多工程事实：

- **每个 `.cpp` 会被单独处理**
- **头文件会被反复展开到不同 `.cpp` 中**
- 单个文件编译通过，不代表整个程序能链接成功
- 第三方库本质上也是“等待被链接进来的目标代码集合”

# 编译的四个阶段

> 传统上，`C++` 构建常被拆成四个阶段：**预处理、编译、汇编、链接**。

## 第一阶段：预处理（Preprocessing）

> 预处理器负责处理所有以 `#` 开头的指令，例如：

- `#include`
- `#define`
- `#if / #ifdef / #ifndef`
- `#pragma`

```cpp
#include <iostream>
#define VALUE 42

int main() {
    std::cout << VALUE << "\n";
}
```

经过预处理后，编译器看到的内容更接近：

- `iostream` 对应头文件内容被插入
- `VALUE` 被替换成 `42`
- 条件编译中不满足条件的代码被移除

### 这个阶段最重要的理解

`#include` 不是其他语言里的“模块导入”，而更像是**文本展开**。 这就是为什么：

- 头文件保护必不可少
- 宏会污染名字空间
- 头文件越重，编译越慢

### 常见问题

- 重复包含导致重复定义
- 宏替换引发难以理解的行为
- 条件编译太多，代码可读性急剧下降

### 典型命令

不同编译器命令不同，但常见工具链一般都支持“只做预处理”：

```bash
g++ -E main.cpp -o main.i
```

得到的 `.i` 文件可以帮助你看清“编译器真正看到的源码是什么”。

## 第二阶段：编译（Compilation）

> 这一步会把 *预处理后的 `C++` 源码* 翻译成更低层的 `汇编代码`。

你可以把它理解成：

- 语法分析
- 语义检查
- 类型检查
- 模板实例化
- 优化
- 生成汇编表示

这一阶段通常会发现你最熟悉的那类错误：

- 语法错误
- 类型不匹配
- 成员不存在
- 模板参数不满足要求
- 访问控制不合法

```cpp
int main() {
    std::string s = 123;
}
```

这类错误属于**编译错误**，链接器还根本没有出场。

### 编译阶段关心什么

编译器在这一步主要关心：

- 这个源文件是否合法
- 所有名字在当前翻译单元里是否“可见”
- 类型是否匹配
- 调用是否合法

> 编译器只需要知道“声明”，通常不需要立刻看到最终定义。  

所以你可以在头文件里声明：

```cpp
int add(int a, int b);
```

只要当前 `.cpp` 看得到这个声明，它就能先通过编译。  
至于这个函数到底有没有真实实现，要等链接阶段才最终确认。

### 典型命令

```bash
g++ -S main.cpp -o main.s
```

这会生成汇编文件 `.s`，方便观察编译器产物。

## 第三阶段：汇编（Assembly）

> 编译器生成的汇编代码还不是最终可被链接的目标文件。汇编器会把这些汇编指令转换为机器码，并打包成目标文件：

- Linux/macOS 常见：`.o`
- Windows 常见：`.obj`

目标文件里通常包含：

- 机器指令
- 符号表
- 重定位信息
- 调试信息（如果开启）

你可以把目标文件理解成：**“已经翻译成机器相关格式，但还没和其他文件拼装完成的零件。”**

> 一个 `.cpp` 通常会产生一个 `.o/.obj`。这些目标文件之后会统一交给链接器。

### 典型命令

```bash
g++ -c main.cpp -o main.o
```

这里的 `-c` 非常常见，表示：

- 编译到目标文件
- 不执行链接

很多大型工程的 *增量构建* 本质上就是：

- 只重新编译改过的 `.cpp`
- 复用未变化的 `.o/.obj`
- 最后统一链接

## 第四阶段：链接（Linking）

> 链接器的任务，是把多个目标文件和库真正拼成一个完整程序。

链接器要做的核心工作包括：

- 合并多个目标文件
- 解析符号引用
- 找到函数和变量的真实定义
- 处理静态库或动态库依赖
- 生成最终可执行文件或共享库

例如：

```cpp
// add.h
int add(int a, int b);

// add.cpp
int add(int a, int b) {
    return a + b;
}

// main.cpp
#include "add.h"

int main() {
    return add(1, 2);
}
```

这里：

- `main.cpp` 编译时只需要看到 `add()` 的声明
- `add.cpp` 编译时生成 `add()` 的定义
- 链接时才把两者真正配对

### 常见链接错误

#### 未定义引用

例如：

- `undefined reference to add`
- `unresolved external symbol add`

通常意味着：

- 只有声明，没有定义
- 对应 `.cpp` 没参与构建
- 忘了链接某个库
- 函数签名不匹配，导致找的不是同一个符号

#### 重复定义

例如：

- `multiple definition of ...`
- `already defined in ...`

通常意味着：

- 某个普通函数定义写进了头文件并被多个 `.cpp` 包含
- 某个全局变量在多个翻译单元里都定义了
- 违反了 `ODR`（单一定义规则）

### 链接阶段最重要的特点

编译器是“逐个翻译单元检查”，链接器才是“站在全局视角组装整个程序”。  
所以经常会出现这种情况：

- 每个 `.cpp` 单独编译都没问题
- 最后链接时报错

这在 `C++` 工程里非常常见，不是例外。

# 翻译单元、符号与链接

> 想真正理解链接，就必须知道 `翻译单元（translation unit）` 和 `符号（symbol）` 在工程里扮演什么角色。

## 翻译单元

一个 `.cpp` 文件加上它递归展开的所有头文件，经过预处理后，就形成了一个翻译单元。

这意味着：

- `main.cpp` 是一个翻译单元
- `logger.cpp` 是另一个翻译单元
- 它们会被分别编译

翻译单元之间默认是隔离的。  
一个 `.cpp` 看不到另一个 `.cpp` 里的局部细节，除非通过头文件声明建立联系。

## 符号

粗略理解，符号就是：

- 函数名
- 全局变量名
- 某些静态对象名
- 编译器和链接器需要识别的命名实体

例如：

```cpp
int add(int a, int b);
int g_value = 0;
```

这里 `add` 和 `g_value` 都会形成符号。

链接器的重要工作之一，就是处理：

- 哪些地方引用了某个符号
- 哪个目标文件真正定义了它

## 声明与定义在链接中的角色

> 声明告诉编译器：*“这个名字合法，你可以先按这个签名生成调用代码。”*

```cpp
int add(int a, int b);
```

> 定义告诉链接器：*“这个符号的真实实现就在这里。”*

```cpp
int add(int a, int b) {
    return a + b;
}
```

所以一句话总结：

- 编译阶段更依赖 **声明**
- 链接阶段最终依赖 **定义**

## 模板与显式实例化

类或函数模板在头文件中给出定义时，每个包含该头的翻译单元均可能 **各自实例化** 一份，从而拉长总编译时间。  
工程上偶见 **显式实例化定义**（在选定 `.cpp` 中写出 `template ...` 的具体实例）配合 **extern template 声明**（在其余翻译单元中禁止再次生成同形实例），以集中生成、缩短冗余编译。

此类写法与构建选项、可见性边界强相关，接手时须在 **头文件与少数源文件** 中对照检索 `extern template` 与显式实例化块，避免重复定义或链接缺失。

# 传统库

> 库（Library）本质上就是 *可复用的已编译代码集合*，供别的程序在构建时或运行时使用。

在 `C++` 里，常见库分为两大类：

- 静态库（Static Library）
- 动态库（Dynamic Library / Shared Library）

此外还有一类 **头文件库（header-only）**，它更接近 *源码分发*，不属于传统二进制库，但工程中非常常见。

库的价值主要体现在：

- 复用代码，避免每个项目都重复实现
- 降低模块之间的源码耦合
- 让体量较大的项目分层构建
- 让第三方能力以标准方式接入

例如：

- 你可以把日志模块编译成一个库
- 多个可执行程序都链接这个库
- 测试程序也可以复用同一套实现

这样项目结构会比“所有源文件直接堆进一个大程序”更清晰。

## 静态库

> 静态库是 *一组目标文件的打包结果*。  
> 链接时，链接器会把程序真正需要的那部分代码拷贝进最终可执行文件。

常见扩展名：
- Windows: `.lib`
- Linux/macOS: `.a`

可以把静态库理解成“零件仓库”：

- 库里放着很多已经编译好的目标代码
- 链接器在链接阶段把 `需要的部分` 取出来
- 最终产物自带这些代码

也就是说，静态库的代码在 **链接时** 进入程序，而不是运行时再去找。

### 优点

- 部署简单，通常只要发一个可执行文件
- 运行时不容易遇到“缺库”问题
- 版本可控，最终产物相对自包含

### 缺点

- 最终二进制通常更大
- 多个程序会各自带一份相同代码
- 更新库后，往往需要重新链接甚至重新发布程序

### 适用场景

- 内部工具
- 部署环境不稳定，尽量减少外部依赖
- 对运行时可用性要求高
- 希望分发简单的单程序场景

## 动态库

> 动态库是 *在运行时加载和使用的二进制库*。

常见扩展名：

- Windows: `.dll`
- Linux: `.so`
- macOS: `.dylib`

在 Windows 上，动态库常常还会配套一个 `.lib` 导入库，供链接阶段使用。

### 工作方式

可以把动态库理解成 **程序运行时再接入的外部部件**：
- 编译和链接时先约定接口
- 运行时由系统加载实际动态库
- 程序调用其中的实现

所以动态库涉及两层问题：

- **链接时** 是否正确引用
- **运行时** 是否真的能找到并加载

### 优点

- 多个程序可共享同一份库文件
- 更新库时不一定需要重新链接整个主程序
- 有利于插件化、模块化部署

### 缺点

- 部署更复杂
- 运行时缺失或版本不匹配时容易失败
- 跨编译器、跨版本 ABI 兼容问题更突出

### 适用场景

- 大型系统的模块化部署
- 插件系统
- 多程序共享同一基础库
- 希望单独升级部分组件

## 如何选择

> 工程里没有绝对正确的选择，关键看部署方式、升级频率、依赖规模和团队环境。

可以用下面这组直觉来判断：

### 更偏向静态库的情况

- 希望部署尽量简单
- 可执行文件自包含更重要
- 依赖数量不多
- 不需要单独升级库

### 更偏向动态库的情况

- 希望多个程序共享同一库
- 需要插件化或热更新思路
- 想让部分组件独立发布
- 大型系统对模块拆分要求更高

### 一个现实建议

对初学者和中小型工程来说，**先把静态链接链路搞明白**，通常比一开始就处理复杂的动态库部署更容易。  
因为动态库除了“怎么编译和链接”，还多了一层“怎么部署和加载”的问题。

# 现代头文件库

> 很多现代 `C++` 库其实不是传统静态库或动态库，而是“头文件库”。

例如：

- 只有头文件
- 使用者直接 `#include`
- 模板和 `inline` 函数在头文件中展开

这类库的特点是：

- 集成简单
- 不需要单独链接库文件
- 但会增加编译时间
- 也更容易把实现细节扩散到整个工程

模板库大量采用这种方式，是因为模板实例化通常需要看到完整定义。

# 注意事项

> 很多构建问题，不是工具太难，而是日常编码习惯不够工程化。

### 1. 区分“编译错误”和“链接错误”

这是最基础也最重要的能力之一。

#### 编译错误常见特征

- 报语法问题
- 报类型不匹配
- 报某个名字在当前作用域不可见
- 报访问控制不允许

说明问题发生在“单个翻译单元内部”。

#### 链接错误常见特征

- `undefined reference`
- `unresolved external symbol`
- `multiple definition`

说明语法层面也许没问题，但全局装配失败了。

#### 运行时加载错误常见特征

- 缺少 `.dll/.so/.dylib`
- 找不到入口点
- 动态库版本不兼容

说明程序已经编译、链接完成，但运行环境不满足要求。

能快速分辨这三类问题，你排查速度会快很多。

### 2. 头文件要自洽

一个高质量头文件应该尽量满足：

- 自己包含自己需要的标准库头
- 能被单独 `#include`
- 不偷偷依赖别的头顺带带入声明

例如，如果头文件里用了 `std::string`，就应直接：

```cpp
#include <string>
```

而不是假设某个别的头已经帮你包含过。

### 3. 头文件里避免放普通定义

这些内容如果直接放头文件，很容易引发链接问题：

- 普通全局变量定义
- 非 `inline` 普通函数定义
- 需要唯一实体但会被多次展开的对象

头文件更适合放：

- 声明
- 模板
- `inline` / 极短函数
- 类型定义

### 4. 减少不必要的 #include

`#include` 过多会带来：

- 编译变慢
- 依赖关系膨胀
- 小改动引发大面积重编译

可行做法包括：

- 能前向声明就前向声明
- 在头文件中少暴露实现细节
- 第三方重型依赖尽量收进 `.cpp`

### 5. 修改头文件要有“波及面意识”

改一个 `.cpp`，往往只影响当前翻译单元。  
改一个公共头文件，可能会触发大量下游文件重新编译。

所以工程里通常会更谨慎地对待：

- 公共接口
- 通用基类
- 基础设施头文件

这些地方一旦变化，构建成本和兼容风险都更高。

### 6. 编译选项要保持一致

在同一个工程里，不同目标如果使用了不一致的关键编译选项，问题会非常隐蔽。  
例如：

- 一个模块用 `-std=c++11`，另一个没开
- Debug / Release 宏不一致
- 不同运行库设置不一致
- ABI 相关选项不一致

这类问题有时不是立刻报错，而是：

- 行为异常
- 链接失败
- 运行时崩溃

所以大型工程里，编译选项通常应由构建系统统一管理，而不是让每个人手工拼命令。

### 7. 区分源码依赖与链接依赖

某个模块可能：

- 编译时需要某个头文件
- 链接时还需要某个库

只满足其中一个通常不够。

典型情况是：

- `#include <fmt/core.h>` 解决了声明可见问题
- 但如果该库不是 header-only，还得在链接阶段加上对应库

很多“我明明 include 了，为什么还报错”的根源就在这里。

### 8. 模板错误常发生在编译阶段，不在链接阶段

模板和普通函数很不一样。  
由于模板实例化往往发生在编译阶段，很多问题会表现为：

- 非常长的编译报错
- 某个实例化链条中断
- 某个模板定义找不到或不完整

这也是为什么模板实现通常放头文件，而不是放到普通 `.cpp`。

### 9. 调试构建与发布构建要分清

Debug 和 Release 往往有明显差别：

- 优化级别不同
- 调试符号有无不同
- 某些断言是否开启
- 未定义行为在优化后更容易暴露

所以不要因为“Debug 能跑”就认为“发布一定安全”。  
在 `C++` 里，很多未定义行为在 Release 下更容易表现得离谱。

### 10. 第三方库接入后要同时验证三件事

只要接入库，就至少要确认：

1. 头文件路径是否正确
2. 链接库是否正确加入
3. 运行时库文件是否能被找到

很多人只验证到第一步，结果：

- 编译通过
- 链接失败

或者：

- 链接通过
- 运行时报缺库

### 11. 学会用构建系统，而不是只记编译命令

手写命令可以帮助理解原理，但真实工程最终还是要依靠：

- `CMake`
- `Makefile`
- IDE 工程文件
- 统一构建脚本

构建系统的作用不只是“帮你少打字”，而是：

- 管理目标之间的依赖
- 统一编译选项
- 管理第三方库
- 支持增量构建
- 支持多平台构建

理解原理之后，把原理交给构建系统去稳定执行，才是工程化做法。

# 一条完整链路示例

下面用一个非常小的例子，把“声明、定义、编译、链接”串起来。

```cpp
// calc.h
#ifndef CALC_H
#define CALC_H

int add(int a, int b);

#endif
```

```cpp
// calc.cpp
#include "calc.h"

int add(int a, int b) {
    return a + b;
}
```

```cpp
// main.cpp
#include "calc.h"
#include <iostream>

int main() {
    std::cout << add(3, 4) << "\n";
    return 0;
}
```

在这个例子里：

1. `main.cpp` 和 `calc.cpp` 各自形成翻译单元
2. 预处理时都会展开 `calc.h`
3. 编译后生成各自的目标文件
4. `main.o` 中记录了对 `add` 的引用
5. `calc.o` 中提供了 `add` 的定义
6. 链接器把它们配对，生成最终程序

如果你删掉 `calc.cpp` 参与构建，通常就会得到：`undefined reference to add`

如果你把 `int add(int a, int b) { ... }` 直接写进普通头文件并被多个 `.cpp` 包含，通常就容易得到： `multiple definition of add`

这个小例子，几乎浓缩了 `C++` 构建中最核心的一组概念。

# 小结

`C++11` 的编译与链接模型，可以浓缩成一句话：

**每个源文件先独立编译成目标文件，再由链接器把这些目标文件和库组装成最终程序。**

如果你只先记住最关键的几件事，可以抓住这些：

- 编译分四阶段：预处理、编译、汇编、链接
- `#include` 是文本展开，不是模块导入
- 编译阶段看声明是否足够，链接阶段看定义是否存在且唯一
- 静态库是“链接时拷贝进程序”，动态库是“运行时加载”
- 很多工程问题不是语法问题，而是依赖、目标、库和构建配置的问题

当你开始用“翻译单元、符号、目标文件、库、链接器”去理解 `C++` 工程时，很多原本看起来神秘的报错，都会变得非常具体、非常可定位。
