# 代码组织

> `C++11` 之后，语言本身已经足够现代，但真正决定项目可维护性的，往往不是“语法会不会写”，而是“代码怎么组织”。

很多初学者写 `C++` 时，容易把注意力放在类、模板、语法糖上；一进入真实工程，就会立刻遇到另一套问题：

- 一个功能应该写进 `.h` 还是 `.cpp`
- 为什么只是 `#include` 多了几个头，编译速度就明显变慢
- 为什么同一个函数写在多个文件里会链接报错
- 第三方库到底是 `复制源码` 还是 `链接库文件`
- 面对一个几万行的工程，应该从哪里开始看

这篇文档讨论的不是单个语法点，而是 **`现代 C++` 工程项目如何把代码拆成一组可以协作的文件与模块**。

你会看到：

- `.h/.cpp` 的职责边界和典型拆分方式
- 预处理、翻译单元、链接之间的关系
- 头文件保护为什么重要，`#pragma once` 与 include guard 怎么选
- 命名空间、类作用域、访问控制在工程中的真正意义
- 第三方库接入的常见方式与注意点
- 阅读复杂工程代码时的实用路线图

# 整体模型

> 读工程前，先把 `C++` 的构建模型想清楚。很多“为什么要这么写”，本质上都来自这个模型。

可以把一个 `C++` 工程粗略理解成三层：

- **接口层**：通常是 `.h/.hpp`，对外声明 *有什么可用*
- **实现层**：通常是 `.cpp`，负责 *具体怎么做*
- **构建层**：由编译器和构建系统把多个源文件编译、链接成最终程序

一个 `.cpp` 文件在编译前会先经历 **预处理**：

- 展开 `#include`
- 展开宏
- 处理条件编译 `#if / #ifdef / #ifndef`

预处理完成后，**每一个 `.cpp` 文件都会独立形成一个翻译单元（translation unit）**。  
编译器分别编译这些翻译单元，最后链接器把它们拼起来。

这件事直接决定了几条工程规则：

- 头文件会被 **文本展开** 到多个 `.cpp` 中，所以头文件必须尽量稳定、简洁、可重复包含
- 头文件里放错定义，可能导致 `重复定义`
- **`.cpp` 之间默认互相看不见，必须通过头文件声明来建立连接**
- `#include` 越多，一个翻译单元就越重，编译速度越慢

# 文件组织

在一个典型 `现代C++` 工程里，你通常会见到这些文件：

- `.h` / `.hpp`：头文件，放声明、类型定义、常量、内联函数、模板
- `.cpp`：源文件，放函数实现、类成员定义、内部辅助逻辑
- 构建文件：如 `CMakeLists.txt`、`Makefile`、`BUILD` 等
- 第三方目录：如 `third_party/`、`external/`、`vendor/`
- 测试目录：如 `tests/`
- 示例或工具目录：如 `examples/`、`tools/`

并不是每个项目都严格遵守同一套目录命名，但整体思想是类似的：  
**让“公开接口、内部实现、依赖、测试、构建脚本”彼此分层。**

### 一个常见目录结构

```text
project/
├── include/
│   └── net/
│       ├── client.h
│       └── server.h
├── src/
│   ├── client.cpp
│   ├── server.cpp
│   └── main.cpp
├── tests/
│   └── client_test.cpp
├── third_party/
├── CMakeLists.txt
└── README.md
```

这种布局背后的好处很直接：

- `include/` 放“要暴露给别人用的接口”
- `src/` 放“不应该随便被外部直接依赖的实现”
- `tests/` 放测试代码，避免和业务代码混在一起
- `third_party/` 单独管理外部依赖，避免污染主代码

对于小项目，不一定要把目录切得这么细；  
但只要项目开始增长，**按职责拆目录** 几乎总比 “所有文件堆一层” 更容易维护。

# 头文件/源文件

一句话记忆：**头文件回答 `你能怎么用我`，源文件回答 `我是怎么实现的`。**

> 头文件适合放：
> - 类、结构体、枚举的定义
> - 函数声明
> - 常量声明
> - 模板定义
> - 必须内联的短小函数
> - 其他模块需要看到的类型别名

```cpp
// math_utils.h
#ifndef MATH_UTILS_H
#define MATH_UTILS_H

int add(int a, int b);
int sub(int a, int b);

#endif
```

这里头文件只描述一件事：**外部代码可以调用 `add()` 和 `sub()`，但暂时不关心实现细节。**

> 源文件适合放：
> - 头文件中声明过的函数定义
> - 类成员函数的非内联实现
> - 文件内部使用的辅助函数
> - 不希望暴露给外部的局部常量、局部类型、内部工具逻辑

```cpp
// math_utils.cpp
#include "math_utils.h"

int add(int a, int b) {
    return a + b;
}

int sub(int a, int b) {
    return a - b;
}
```

## 二者如何协同

`.h` 和 `.cpp` 的协作可以理解为：

1. 头文件先把“接口契约”声明出来
2. 源文件再去提供真实定义
3. 使用者只需要 `#include` 头文件并调用接口
4. **链接阶段，链接器把“调用点”和“定义”匹配起来**

如果只有声明没有定义，链接时常见报错会是：

- `undefined reference`
- `unresolved external symbol`

如果定义出现了多份，常见报错会是：

- `multiple definition`
- `LNK2005 already defined`

> 从工程角度看，`.h/.cpp` 的拆分并不只是“文件习惯”，而是在配合编译器和链接器工作。

## 类的典型拆分方式

真实项目里最常见的是“类声明在头文件，成员定义在源文件”：

```cpp
// logger.h
#ifndef LOGGER_H
#define LOGGER_H

#include <string>

class Logger {
public:
    explicit Logger(const std::string& name);
    void log(const std::string& message);
private:
    std::string name_;
};

#endif
```

```cpp
// logger.cpp
#include "logger.h"
#include <iostream>

Logger::Logger(const std::string& name) : name_(name) {}

void Logger::log(const std::string& message) {
    std::cout << "[" << name_ << "] " << message << "\n";
}
```

这种拆分有几个工程收益：

- 头文件更稳定，改实现不一定触发所有使用者重新编译
- 用户只依赖接口（头文件），不必看到实现细节
- *源文件可以自由引入实现依赖，不把这些依赖传播给所有包含者*

## 不适合拆分的场景

有几类代码不能简单地“声明在 `.h`，定义在 `.cpp`”：

### 1. 模板

模板在实例化时必须看到完整定义，因此很多模板代码直接写在头文件里：

```cpp
template <typename T>
T max_value(const T& a, const T& b) {
    return a < b ? b : a;
}
```

如果只在头文件里写声明，而把模板定义放到普通 `.cpp` 中，其他翻译单元通常无法实例化它。

### 2. inline 函数

`inline` 的重点不是“编译器一定内联”，而是 **允许函数定义出现在多个翻译单元中而不违反 ODR**。  
因此它通常也写在头文件中。

### 3. 类内定义的短函数

写在类定义内部的成员函数默认具有 `inline` 属性，适合极短、极稳定的逻辑：

```cpp
class Counter {
public:
    int value() const { return value_; }

private:
    int value_ = 0;
};
```

## 实用原则

### 原则 1：头文件尽量只暴露必要信息

如果某个实现细节不需要让外部知道，就不要放进头文件。  
头文件越“胖”，依赖传播越严重，编译越慢，耦合越强。

### 原则 2：源文件包含自己的头文件

一个很重要的习惯：

```cpp
#include "logger.h"
```

把对应头文件放在 `.cpp` 的第一位包含，能尽早发现头文件是否“自洽”。  
如果一个头文件离开某些“顺带包含”的依赖就编不过，说明它组织得不够干净。

### 原则 3：一个头文件应该能单独被包含

理想状态下，你应该可以在任意 `.cpp` 里直接：

```cpp
#include "logger.h"
```

并成功编译。  
这意味着头文件应该自行包含它所需要的标准库头，而不是偷偷依赖别的文件先替它包含。

### 原则 4：接口稳定，依赖内收

例如，类的私有成员如果直接写成某个重型第三方类型，那么所有包含这个头文件的地方都要看到那个第三方头。  
这会让耦合和编译时间一起上升。

这也是很多大型工程偏爱：

- 前向声明
- PImpl（pointer to implementation）
- 抽象接口 + 工厂函数

这些模式的原因之一。

# include 规则

`#include` 支持两种写法，`""` 和 `<>`，二者区别明显，主要体现在 `搜索顺序` 和 `代码语义` 上。​

> `""`：用于包含​​你自己项目内部​​的、本地的头文件；搜索顺序：
>   1. 在包含该 #include 指令的​​当前源文件所在的目录​​中查找。
>   2. 如果在当前目录没找到，编译器会去构建系统（如 CMake、Makefile）通过 -I 参数指定的包含目录中查找。
>   3. ​最后​​，如果还没找到，再去系统的标准库目录（如 /usr/include）中查找。

> `<>`：用于包含​​标准库​​、​​系统头文件​​以及​​第三方外部依赖库​​的头文件；搜索顺序：
>   1. 直接去构建系统通过 -I 参数指定的包含目录中查找。
>   2. 去系统的标准库目录中查找。
>   3. 它​ **​不会** ​​优先搜索当前源文件所在的目录。

这不是在说 `<>` 只能包含系统头、`" "` 只能包含项目头，**两者都能包含任意文件**。  

> 如果某个目录里存在同名文件，**先被搜索到的那个就会生效**。  

这也是“同名头文件阴影（shadowing）”问题的根源：  
项目里不小心放了一个 `string` / `windows.h` / `stdio.h` 之类的同名文件，会导致非常离谱的编译错误。

> GCC/Clang 把搜索路径大致分成几类（不同版本/平台细节略有差异，但足够用于工程判断）：

- `-I<dir>`：添加 “普通 include 路径”。会参与 `""` 与 `<>` 的搜索。
- `-iquote <dir>`：添加 “只给 `""` 用”的路径（不会给 `<>` 用）。
- `-isystem <dir>`：添加 “系统 include 路径”。常用于第三方依赖，很多警告会对系统头更宽松。
- **内置系统目录**：例如标准库、编译器自带 headers、平台 SDK 等。

排查时常用：

- **打印 include 树**：`-H`（会输出每个被包含头文件的路径与层级）
- **打印/查看搜索路径**：可用 `-v`（通常会在输出里列出头文件搜索目录）

> 在现代 CMake 里，头文件查找路径通常来自：

- `target_include_directories(<tgt> ...)`：给目标添加 include dirs
  - `PRIVATE`：只影响本目标的编译
  - `PUBLIC`：影响本目标 + 依赖它的目标
  - `INTERFACE`：只影响依赖它的目标（常用于 header-only 库）

> 实战建议:  
- **项目内头文件**：优先用 `""`，建议 `相对工程根路径`，例如 `#include "net/client.h"`，不要依赖 *碰巧当前目录能搜到*。
- **标准库/系统/第三方库头**：优先用 `<>`，例如 `#include <vector>`、`#include <fmt/format.h>`。
- **避免相对路径上跳**：例如 `#include "../common/x.h"` 往往会让重构和跨平台构建更痛苦

# 从预处理到链接

> `C++` 工程组织的很多“规矩”，本质上都来自预处理和链接规则。

> `#include` 不是导入模块，而是文本插入

很多初学者会把 `#include`误解成其他语言里的 `import`，实际上它更接近 **把那个文件的文本原封不动插进来**。

例如：

```cpp
// a.cpp
#include "logger.h"
```

预处理后更像是：

```cpp
// a.cpp
class Logger {
public:
    explicit Logger(const std::string& name);
    void log(const std::string& message);

private:
    std::string name_;
};
```

这就是为什么：

- 重复包含会出问题
- 头文件必须有保护
- 头文件里写宏、全局变量、非 `inline` 函数定义时要非常小心

## 声明与定义

`声明` 告诉编译器：**这个名字存在，它的类型和签名是什么。**

```cpp
int add(int a, int b);
class Logger;
extern int g_value;
```

`定义` 告诉编译器：**这个实体真正占据了存储或给出了完整实现。**

```cpp
int add(int a, int b) {
    return a + b;
}

int g_value = 42;
```

工程中最常见的模式就是：

- 在头文件放声明
- 在源文件放定义

这样多个翻译单元都能看见同一套接口，但最终只有一份真实实现。

## ODR：单一定义规则

> `ODR`（One Definition Rule）是 `C++` 工程里非常重要的一条规则。粗略理解就是：
> - 一个实体可以被多次声明
> - 但某些实体只能有一个有效定义

违反 `ODR` 的后果很典型：

- 链接报错
- 更糟时，编译通过但行为未定义

高频出错点包括：

- 在头文件中定义普通全局变量
- 在头文件中定义非 `inline` 普通函数
- 同一个类的定义在不同文件里出现不一致版本

例如下面写法就很危险：

```cpp
// bad.h
int g_counter = 0;
```

任何包含 `bad.h` 的 `.cpp` 都会各自产生一份 `g_counter` 定义，最后链接极易出问题。

更合理的写法是：

```cpp
// counter.h
#ifndef COUNTER_H
#define COUNTER_H

extern int g_counter;

#endif
```

```cpp
// counter.cpp
#include "counter.h"

int g_counter = 0;
```

# 预处理与头文件保护

> 头文件可能会被间接包含很多次，所以 `可重复包含而不出错` 是基本要求。

考虑这种情况：

- `main.cpp` 包含了 `a.h`
- `a.h` 又包含了 `b.h`
- `main.cpp` 自己也直接包含了 `b.h`

如果 `b.h` 没有保护，预处理后它的内容就会重复出现，类、函数声明甚至宏都可能重复展开，轻则编译报错，重则引入诡异行为。

## include guard

最传统、最通用的写法是：

```cpp
#ifndef LOGGER_H
#define LOGGER_H

class Logger {
public:
    void log();
};

#endif
```

原理很简单：

- 第一次包含时，`LOGGER_H` 还没定义，头文件内容会生效
- 同一翻译单元后续再次包含时，宏已经定义，这段内容会被跳过

这是最经典的头文件保护方式，兼容性也最好。

## #pragma once

另一种更简洁的写法：

```cpp
#pragma once

class Logger {
public:
    void log();
};
```

它的优点是：

- 写法更短
- 不需要自己维护宏名
- 少掉宏名冲突的可能

它的缺点是：*严格来说不是最早期标准的一部分，而是广泛支持的编译器扩展*

> 现代主流编译器基本都支持它，所以很多工程会直接使用 `#pragma once`。  

如果你处在跨平台、老编译器、工具链非常保守的环境里，`include guard` 仍然是最稳妥的方式。

## 宏与预处理的边界

预处理器除了 `#include`，还会处理宏：

```cpp
#define SQUARE(x) ((x) * (x))
```

> 宏在工程中不是不能用，但要非常克制。原因是：
> - 它不遵守正常的作用域规则
> - 它只是文本替换，容易产生副作用
> - 调试体验差
> - 很容易与第三方库或系统头中的宏冲突

例如：

```cpp
int a = 3;
int b = SQUARE(a + 1); // 展开后是 ((a + 1) * (a + 1))
```

这个例子看似安全，但如果宏写得不好，副作用会很难排查。  
在 `C++11` 工程里，很多宏都可以用更安全的方式替代：

- 常量宏 -> `const` / `constexpr`
- 函数式宏 -> `inline` 函数 / 模板函数
- 条件开关 -> 构建配置或局部条件编译

## 条件编译

条件编译仍然很常见，尤其在这些场景里：

- 平台差异：Windows / Linux / macOS
- Debug / Release 行为切换
- 第三方库是否可用
- 某些实验性功能开关

```cpp
#ifdef _WIN32
// Windows 特定实现
#else
// POSIX 实现
#endif
```

但要注意：**条件编译越多，代码就越像“同时维护多份程序”。**  
它是必要工具，但过量使用会显著提高理解成本。

## 前向声明

> 如果头文件里只需要 *知道某个类型存在，不需要它的完整定义*，可以使用前向声明：

```cpp
class Logger;

class Service {
public:
    void setLogger(Logger* logger);
private:
    Logger* logger_;
};
```

这样 `service.h` 就不必直接包含 `logger.h`。这会带来两个收益：
- 减少编译依赖
- 避免头文件之间相互包含

但前向声明也有边界。以下情况通常需要完整定义，而不仅仅是声明：

- 按值持有对象成员
- 继承某个类型
- 调用该类型的成员函数
- 使用 `sizeof`

所以经验上可以记住：

- **指针 / 引用 常适合前向声明**
- **按值成员 往往需要完整头文件**

# 作用域与访问控制

> 工程里的“封装”不只是面向对象语法，更是在控制名字可见范围、限制耦合面、缩小出错半径。

## 作用域层次

在 `C++` 项目中，你会频繁遇到这些作用域：

- 块作用域：花括号内部的局部变量
- 函数作用域：函数参数与函数体内部名字
- 类作用域：类成员
- 命名空间作用域：模块级名字组织
- 文件级可见性：通过内部链接限制只在当前翻译单元可见

### 块作用域

```cpp
void f() {
    int x = 1;
    {
        int y = 2;
    }
    // y 在这里不可见
}
```

块作用域主要解决的是：

- 生命周期控制
- 临时变量污染减少
- 逻辑局部化

### 命名空间作用域

工程代码几乎都应该进入命名空间：

```cpp
namespace net {

class Client {
public:
    void connect();
};

} // namespace net
```

命名空间的意义不是“写起来麻烦一点”，而是：

- 避免全局命名冲突
- 把同一模块的概念组织在一起
- 为体量较大项目的分层提供天然边界

如果一个复杂工程没有命名空间，很快就会遇到函数名、类名、常量名彼此撞车的问题。

### 类作用域

类把数据和行为收进一个类型边界内，也顺便定义了成员可见范围。

```cpp
class Account {
public:
    void deposit(int money);
    int balance() const;
private:
    int balance_ = 0;
};
```

这里 `balance_` 只能由类内部直接访问，外部必须通过接口间接操作。

## public/protected/ private

很多教材会把访问控制讲成一句简单规则：

- `public`：外部可见
- `protected`：子类可见
- `private`：仅类内部可见

但在工程实践里，更重要的是下面这个思路：

### public 是承诺

一旦某个成员进入 `public` 接口，你就相当于对使用者做了承诺：**别人可以依赖它，我今后改动它要非常小心。**

所以 `public` 应该尽量少而稳定。

### private 是缓冲区

`private` 允许你重构内部实现而不影响外部调用者。  
工程上真正的可维护性，很多时候就来自这种 *外部接口稳定，内部可自由调整* 的空间。

### protected 要谨慎

`protected` 常用于给子类留扩展点，但它也会扩大耦合面。  
一旦子类开始深度依赖基类内部细节，后续重构就会很痛苦。

很多现代工程宁可：
- 组合优于继承
- 暴露清晰的 `public` 扩展接口
- 少让子类直接摸内部状态

也不愿意滥用 `protected`。

## 匿名命名空间

如果某个函数或对象只该在 **当前 `.cpp` 文件内** 使用，常见做法是放进匿名命名空间：

```cpp
namespace {

int parse_port(const std::string& text) {
    return std::stoi(text);
}

} // namespace
```

这样它只在当前翻译单元可见，外部无法链接到它。这非常适合：

- 文件内部辅助函数
- 仅本文件使用的常量
- 某些局部工具对象

在 `C++` 工程中，这类“内部实现”不要轻易暴露到全局命名空间或头文件里。

### static 与文件作用域

在较老的写法中，也会看到：

```cpp
static int helper();
```

放在文件作用域上时，它同样表示内部链接。  
现代风格里更常见的是匿名命名空间，因为它也适用于类型、对象和函数，表达更统一。

## 实参依赖查找概要

对函数调用 `f(x)`，除常规名字查找外，编译器还会在 **实参类型所关联的命名空间** 中查找候选函数，该机制称为 **实参依赖查找（ADL）**，亦称 **Koenig 查找**。

典型影响包括：对用户定义类型调用 `swap(a,b)` 时，常见写法为 `using std::swap; swap(a,b);`，使 **特化于该类型的非成员 `swap`** 经 ADL 进入候选集，同时保留对 `std::swap` 的回退。  
在自定义命名空间中重载与标准库同名的自由函数时，须注意 **无意经 ADL 参与其它调用点** 的风险；公共接口宜控制命名与参数设计，避免与标准算法名偶然冲突。

## 实用规则

工程组织上有一条非常实用的规则：**名字的可见范围应尽可能小。**

- 局部变量不要提到更大作用域
- 只在文件内使用的函数不要进头文件
- 不该被外部依赖的成员不要放 `public`
- 不必全局共享的对象不要做全局变量

> 范围越大，未来被误用的概率就越高。

# 头文件依赖管理

> 复杂工程不是 *代码太多*，而是 *依赖关系太乱*。

### 包含什么，为什么重要

每多一个 `#include`，当前翻译单元要处理的代码就会更多。  
如果头文件 A 包含头文件 B，B 又包含 C、D、E，那么所有包含 A 的 `.cpp` 都会一起吞下这整串依赖。

这会带来三类成本：

- 编译速度变慢
- 头文件耦合增强
- 改一个底层头文件，触发大量无关文件重编译

### include what you use

一个很重要的原则是：**谁使用某个声明，谁显式包含对应头文件。**

不要依赖 *反正别的头已经顺带包含了*。  
因为一旦上游头文件调整，你的代码就可能突然编不过。

例如，如果你在头文件中使用了 `std::string`，就应该直接：

```cpp
#include <string>
```

而不是期待某个别的头文件顺便帮你带上。

### 避免循环包含

错误示例：

- `a.h` 包含 `b.h`
- `b.h` 又包含 `a.h`

即使有头文件保护，这种写法也会让类型依赖关系变得脆弱，尤其当双方都按值持有对方成员时，往往直接无法成立。

解决思路通常是：

- 能前向声明就前向声明
- 能用指针/引用表示依赖，就避免按值强耦合
- 重新审视设计，看看是否职责拆分不合理

> 很多 `头文件打架` 的问题，最后都 *不是 include 技巧问题，而是设计层面的循环依赖*。

# 小结

`C++11` 背景下的现代工程代码组织，可以浓缩成一句话：

**用清晰的接口边界、受控的依赖关系和合理的可见性，把“可编译”进一步提升为“可维护”。**

如果你只记住最关键的几件事，可以先抓住这些：

- `.h` 放接口，`.cpp` 放实现
- `#include` 是文本展开，所以头文件必须受保护、尽量轻量
- 作用域和访问控制的本质，是缩小耦合面和误用范围
- 第三方库要接入，但不要让它的细节污染整个工程
- 阅读复杂项目时，先看结构和主流程，再钻进局部实现

当你开始用“翻译单元、依赖传播、接口稳定性、可见性边界”去思考 `C++` 项目时，你就已经不再只是“会写代码”，而是在真正进入工程化开发。
