# 测试与验证

很多其它语言开发者进入 `C++` 项目时，首先想到的是“怎么编译起来”；  
但真正参与 **团队维护的** `C++` 项目后，你很快会发现另一件事同样重要：

> 你不只是要把它编出来，还要有办法确认“我改的东西没有把别的地方弄坏”。

这就是测试和验证的价值。

在 `C++11` 工程里，测试不只是“有空再补”，它经常直接决定你是否敢改代码。

----

# 测试价值

`C++` 的很多问题并不会用友好的异常把你温柔提醒出来。  
你更可能遇到的是：

- 链接失败
- 运行时崩溃
- 未定义行为
- 只在某些输入或某些平台下出现的问题

所以在 `C++` 里，测试的意义通常不只是验证业务逻辑，还包括：

- 固定行为边界
- 保护重构
- 缩小排查范围
- 帮助新人理解模块应该怎样被使用

很多时候，测试本身就是阅读工程代码最快的入口之一。

----

# 验证方式

## 手工验证

最简单的方式，例如：

- 编译运行一个小程序
- 手动输入几组参数
- 看输出是否符合预期

它对学习阶段很有价值，但不适合作为长期回归保障。

## 自动测试

把输入、调用和期望结果写成可以重复运行的程序。

这类测试更适合：

- 保证重构安全
- 提供回归保护
- 进入 CI

## 工具验证

比如：

- `ctest`
- Sanitizer
- 静态分析

这些东西不一定长得像“断言式单元测试”，但它们同样属于验证体系。

----

# 单元测试

一个好的单元测试通常回答的是：

- 这个模块在正常输入下应该怎么表现
- 在边界条件下应该怎么表现
- 在异常输入下应该怎么失败

例如一个解析函数：

```cpp
int parse_port(const std::string& text);
```

对应测试关注的可能是：

- `"8080"` 是否解析成功
- `"0"` 是否允许
- 空串是否失败
- 超出范围时如何处理

这类测试会比“通读整个实现”更快地告诉你模块边界。

----

# 工程作用

因为 **多人维护、模块较多** 的代码库，其真正成本往往不是“写一个功能”，而是：

> 你不知道改动会不会影响到很远的地方。

在 `C++` 项目里，这种担心会更强，因为你不仅要考虑：

- 业务逻辑有没有变
- 接口契约有没有变

还要考虑：

- 生命周期有没有被破坏
- 资源是否还安全释放
- 多线程是否还能成立
- 链接和依赖关系有没有被改坏

所以测试在这类工程里不是附属品，而是“敢改代码”的前提之一。

----

# 最小示例

即使不用任何测试框架，你也可以先写最朴素的验证程序：

```cpp
#include <cassert>

int add(int a, int b) {
    return a + b;
}

int main() {
    assert(add(1, 2) == 3);
    assert(add(-1, 1) == 0);
}
```

这当然很基础，但它已经比“肉眼看输出感觉没问题”更可靠。

它至少把“输入”和“期望结果”固定了下来。

----

# GoogleTest

很多 `C++` 项目会使用 `GoogleTest`（通常简称 `gtest`）做单元测试。

最常见的好处包括：

- 断言表达清楚
- 测试组织结构清楚
- 易于批量运行
- 更容易接入 `ctest` 和 CI

一个典型例子：

```cpp
#include <gtest/gtest.h>

int add(int a, int b) {
    return a + b;
}

TEST(AddTest, BasicCase) {
    EXPECT_EQ(add(1, 2), 3);
    EXPECT_EQ(add(-1, 1), 0);
}
```

你现在不必先把所有宏背熟，但至少要知道：

- `TEST` 在定义测试用例
- `EXPECT_EQ` 在表达期望值

----

# CTest

`ctest` 是 `CMake` 生态里最常见的测试运行入口之一。

如果项目已经在 `CMakeLists.txt` 里接好了测试，通常可以直接运行：

```bash
ctest --test-dir build
```

这意味着：

- 你不需要记住每个测试可执行文件路径
- 测试可以被统一收集和执行
- CI 也更容易直接调用

所以接手项目时，如果你先学会：

- 怎么构建测试
- 怎么运行 `ctest`
- 怎么看失败输出

通常收益非常高。

----

# CMake 接入

```cmake
enable_testing()

add_executable(math_test math_test.cpp)
target_link_libraries(math_test PRIVATE mylib gtest_main)

add_test(NAME math_test COMMAND math_test)
```

这里做了三件事：

1. 打开测试支持
2. 编译测试程序
3. 把测试程序注册给 `ctest`

从工程角度看，这意味着测试已经成为构建系统的一部分，而不是“谁想起来谁手工跑一下”。

----

# 测试内容

在 `C++` 里，很多重要问题并不只体现在一个返回值上。

你还经常需要测试：

- 对象状态变化
- 容器内容变化
- 异常是否被抛出
- 错误路径是否走对
- 某个边界条件下是否会崩

例如异常：

```cpp
EXPECT_THROW(parse_port("abc"), std::invalid_argument);
```

例如状态：

```cpp
EXPECT_TRUE(queue.empty());
```

----

# 补测时机

不是所有代码都值得无脑加测试，但下面这些情况通常很值：

- 业务规则明显
- 边界条件多
- 之前出过 bug
- 重构风险高
- 模块对外接口清晰

反过来，如果只是一个非常薄的转发层、打印层、样板胶水层，测试价值可能没有你想得那么高。

----

# 项目接手

拿到一个陌生项目时，测试有几个非常实用的用途：

## 测试目录

常见位置包括：

- `tests/`
- `test/`
- `unittest/`

## 跑通测试

这是最重要的一步。  
如果连测试都跑不起来，你对项目的“安全修改范围”就会非常模糊。

## 先读失败

如果某个测试已经失败，它经常能比实现更快告诉你：

- 预期行为是什么
- 当前偏差在哪里

## 行为文档

很多时候测试代码不是在告诉你“这行实现怎么写”，而是在告诉你：

> 作者希望这个模块如何被调用、如何表现、哪些行为被视为稳定契约。

----

# 实用习惯

## 回归测试

如果你正在修一个 bug，通常最值的是：

- 先写一个能复现 bug 的测试
- 再修代码
- 最后确认测试通过

这能把“以后别再坏回去”固定下来。

## 公共接口

大多数时候，测试更适合围绕：

- 类的公共方法
- 模块对外函数
- 对外暴露的行为

而不是死盯私有实现细节。

## 边界场景

例如：

- 空输入
- 单元素
- 超长输入
- 非法输入
- 重复调用

这些地方在 `C++` 项目里尤其容易暴露问题。

----

# 验证体系

测试不能替代所有事情。

在 `C++` 工程里，比较稳妥的组合通常是：

- 单元测试：验证行为和边界
- `ctest`：统一执行测试
- Sanitizer：补充发现内存和未定义行为问题
- 编译警告：在编译期暴露一批问题

也就是说，测试体系通常不是单件工具，而是一组互相补位的手段。

----

# 推荐实践

- 测试要正式纳入构建系统，而不是停留在手工脚本层。
- 先学会运行项目现有测试，再谈新增测试。
- 修 bug 时优先考虑补一个能复现问题的测试。
- 测试优先围绕公开行为和边界条件，而不是私有实现细节。
- 把测试当作项目行为文档的一部分来阅读。

----

# 小结

对参与 `C++11` **团队项目或长期维护代码库** 的人来说，测试的价值并不只是“写得更规范”，而是：

- 给修改提供护栏
- 给重构提供信心
- 给阅读提供行为地图

你不一定一开始就要精通某个测试框架，但至少应该尽快掌握三件事：

1. 怎么把项目测试跑起来
2. 怎么读测试里的行为意图
3. 怎么为一个真实 bug 补一个最小回归测试
