# 后续版本概览

这套文档的主线会始终围绕 `C++11`。

但当你真正进入项目后，很可能会碰到这样一种情况：

- 项目整体仍按 `C++11` 思维组织
- 但局部代码里已经掺进了一些 `C++14`、`C++17` 甚至 `C++20` 写法

这时最需要的往往不是立刻系统补完更高版本，而是先知道：

> 这些新写法大概是什么，它们解决了什么问题，和 `C++11` 相比多了什么。

这篇文章就是为这个目的准备的。

----

# 学习原则

如果你现在的目标是接手基于 `C++11` 编译的大型项目，那么更合理的学习顺序通常是：

1. 先把 `C++11` 主线学扎实
2. 再把更高版本特性当作“识别扩展”来理解
3. 不要让更高版本的语法糖冲散你的 `C++11` 基础模型

也就是说：

- 这篇文章帮助你“看懂”
- 不要求你立刻把项目全面改写成更高版本风格

----

# C++14

很多人会把 `C++14` 理解成：

> 在 `C++11` 基础上把一些高频痛点补顺了

常见感知最明显的点包括：

## make_unique

在 `C++11` 里你常见的是：

```cpp
std::unique_ptr<User> user(new User());
```

到 `C++14`，更常见的是：

```cpp
auto user = std::make_unique<User>();
```

它更简洁，也更不容易在复杂表达式里写出异常安全问题。

## 泛型 lambda

`C++11` 里 lambda 参数类型要显式写：

```cpp
auto print = [](int x) {
    std::cout << x << '\n';
};
```

到 `C++14` 可以写：

```cpp
auto print = [](auto x) {
    std::cout << x << '\n';
};
```

## 返回值推导

某些函数可以更自然地推导返回类型，不需要像 `C++11` 那样频繁写尾置返回类型。

----

# C++17

如果一个项目带一点 `C++17` 味道，你很可能最先碰到这些：

## 结构化绑定

```cpp
auto [key, value] = item;
```

它让遍历 `map`、处理 `pair` 时更自然。

## if constexpr

它让模板代码里的条件分支更容易读。

## optional

它常用来表达：

> 这里可能有值，也可能没有值

这比某些旧风格的“返回 bool 再配输出参数”更直观。

## string_view

它表达的是一个只读字符串视图，而不是拥有字符串本身。  
读项目代码时要特别注意：

- 它通常不拥有底层数据
- 生命周期仍然要靠别的对象保证

## variant

它适合表达“这里是多种类型之一”，经常被用来替代一部分手工标签联合体风格代码。

----

# C++20

`C++20` 的内容很多，但对读项目代码来说，最常见的感知点通常有这些：

## 协程

你会看到：

- `co_await`
- `co_return`
- `co_yield`

这通常意味着项目在做更现代的异步抽象。

## concepts

模板约束会更显式，报错也可能更友好。

## ranges

标准库算法和容器操作会出现新的组织方式。

如果你当前主线还是 `C++11`，对这些内容先做到“能识别出它在干什么”就够了。

----

# 版本关系

不要把更高版本特性理解成“彻底推翻旧模型”，更稳妥的看法通常是：

- `C++14`：让 `C++11` 更顺手
- `C++17`：让很多常见表达更自然
- `C++20`：继续增强抽象能力和标准库表达力

也就是说，它们大多是在 `C++11` 已经建立起来的主线上继续往前推。

如果你还没把这些主线学稳：

- 值语义
- 引用语义
- 所有权
- RAII
- 移动语义
- 标准库容器和算法
- 类设计和接口边界

那么先沉迷更高版本的新语法，收益通常不大。

----

# 项目阅读

如果你接手的是以 `C++11` 为主的项目，但局部代码已经用了更高版本能力，建议这样处理：

## 编译标准

不要只看代码长得像什么，更要看构建系统实际用了什么标准。

例如：

- `-std=c++11`
- `-std=c++14`
- `-std=c++17`

## 先看语义

看到 `make_unique`、结构化绑定、`optional`，先搞清它们在表达什么，不要立刻冲动地大范围风格迁移。

## 保持一致

如果一个模块明确还是按 `C++11` 标准维护，就不要随手往里加只能在更高标准编译的特性。

----

# 新旧对照

## C++11 写法

```cpp
std::unique_ptr<User> user(new User());
std::pair<int, std::string> item(1, "hello");
int value = condition ? 1 : 2;
```

## 更高版本

```cpp
auto user = std::make_unique<User>();
auto [id, text] = item;
if constexpr (std::is_integral<T>::value) {
}
```

读到这些代码时，最重要的不是“这个语法我背过没有”，而是：

- 它在解决什么原来的痛点
- 和 `C++11` 主线是否兼容
- 当前项目是否真的允许使用它

----

# 推荐实践

- 先把 `C++11` 主线学稳，再系统补更高版本。
- 读项目代码时先确认构建标准，不要只凭语法猜。
- 对更高版本特性先做到“能识别、能查阅、能理解语义”。
- 如果项目以 `C++11` 为边界，新增代码尽量保持这一边界，不要随意越界。

----

# 小结

这篇文章的重点不是让你立刻进入 `C++14/17/20` 深水区，而是帮你建立一个更稳的判断：

- 主线仍然是 `C++11`
- 更高版本特性是扩展，不是替代基础
- 先看懂项目里出现的新写法，再决定是否系统补课

只要这个边界感清楚，你就不太会在“学 `C++11`”的阶段，被更高版本的局部语法打乱节奏。
