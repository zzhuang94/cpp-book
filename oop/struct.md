# 结构体

结构体是 `C++` 中非常基础但又容易被低估的类型工具。  
很多初学者会把 `struct` 理解成“比 `class` 简陋一点的写法”，但在现代 `C++` 里，它更像是一种设计信号：

> **这个类型主要是轻量数据载体，而不是强调强封装的对象。**

`struct` 和 `class` 在语法能力上几乎一样，两者的核心区别，更多在于**默认表达的设计意图**，而不是功能强弱。

----

# 基本用法

```cpp
#include <iostream>
#include <string>

struct User {
    int id;
    std::string name;
};

int main() {
    User user{1, "alice"};
    std::cout << user.id << ", " << user.name << std::endl;
    return 0;
}
```

这个例子最适合用来理解 `struct` 的默认风格：

- 数据成员直接公开
- 使用者可以直接读写字段
- 类型更像“打包在一起的数据”

如果一个类型的主要职责就是承载几项数据，这种写法通常非常自然。

> 不要把它理解成“过时的 C 风格遗留物”

现代 `C++` 里，`struct` 仍然非常常见，例如：

- 配置对象
- 返回值聚合
- 坐标、颜色、矩形这类小型值类型
- 简单 DTO 或消息对象

所以 `struct` 并不是“不会面向对象时才用”的语法，而是**在合适场景下更清晰的默认选择**。

----

# 聚合初始化

`C++11` 中，很多简单结构体可以直接用花括号初始化：

```cpp
struct Point {
    int x;
    int y;
};

Point p{10, 20};
```

这种写法非常直观，适合轻量数据载体。  
聚合初始化和 `struct` 的设计意图高度一致：

- 字段公开
- 结构简单
- 使用者一眼就能看出对象的内容

对小型值类型来说，这种“所见即所得”的初始化体验非常好。

不过要注意：本书聚焦 `C++11`，在这个标准下，聚合类型有一些限制。  
例如一旦你自定义构造函数，它通常就不再是简单聚合了。

----

# 成员默认值

`C++11` 允许类和结构体成员在定义处提供默认值：

```cpp
#include <string>

struct Config {
    int port = 8080;
    bool debug = false;
    std::string host = "127.0.0.1";
};
```

这能显著减少样板构造代码，也让默认状态更集中。 
像 `Config` 这种类型，重点往往不是封装复杂规则，而是表达“一组默认可用的数据”。  
成员默认值的好处是：

- 默认状态写在字段旁边，容易读
- 少数字段需要修改时，只改对应字段即可
- 不必为了几个默认值专门写一长串构造函数

这类写法很符合配置对象的使用习惯。

----

# 成员函数

`struct` 不只是被动数据容器，也可以拥有行为：

```cpp
#include <iostream>

struct Point {
    int x;
    int y;

    void print() const {
        std::cout << "(" << x << ", " << y << ")" << std::endl;
    }
};
```

!> 这不意味着它就“不再是 struct”

很多人误以为：  
只要一个类型里开始写成员函数，它就应该立刻改成 `class`。  
其实不是。判断关键不在于“有没有函数”，而在于这些函数在做什么：

- 如果只是围绕数据提供自然的小操作，`struct` 完全合理
- 如果开始承担复杂不变量、权限控制、资源所有权管理，`class` 往往更合适

例如下面这些函数通常很适合保留在 `struct` 里：

- `print()`
- `empty()`
- `area()`
- `normalize()`

它们更像是“这个数据类型的自然操作”，而不是复杂业务封装。

----

# 构造函数

`struct` 也完全可以写构造函数：

```cpp
#include <string>

struct User {
    int id;
    std::string name;

    User(int userId, const std::string& userName)
        : id(userId), name(userName) {
    }
};
```

这说明 `struct` 完全可以参与面向对象设计，只是默认访问权限不同。

如果构造函数只是为了：

- 让初始化更方便
- 避免字段漏填
- 提供更自然的创建方式

那么继续使用 `struct` 往往没问题。

但如果构造函数开始承担下面这些职责，就要考虑这个类型是不是已经更像 `class` 了：

- 严格检查输入合法性
- 维护复杂不变量
- 隐藏内部表示
- 接管资源生命周期

也就是说，问题不是“能不能写构造函数”，而是：
> **这个类型是不是已经开始从“数据载体”演变成“有边界的对象”。**

----

# 继承

虽然很多 `struct` 只是轻量数据类型，但语法上它也可以继承：

```cpp
struct Base {
    void print() const {
    }
};

struct Derived : Base {
};
```

由于 `struct` 默认是 `public` 继承，这在表达一些简单扩展关系时会比较顺手。

不过在真实项目里，如果继承层次开始变复杂，读代码的人往往更希望看到明确的抽象边界。  
> 所以一旦进入复杂继承、多态、封装设计，很多代码还是会更偏向 `class`。

----

# 综合示例

下面这个例子把 `struct` 最常见的几种用途放在一起：

```cpp
#include <iostream>
#include <string>

struct Book {
    std::string title = "unknown";
    int pages = 0;

    bool empty() const {
        return title.empty() || pages == 0;
    }

    void print() const {
        std::cout << title << ", " << pages << std::endl;
    }
};

int main() {
    Book b1;

    // 本书以 C++11 为背景。这个 Book 带有默认成员初始值，
    // 因此这里不再把它当作最简单的聚合初始化示例来使用。
    Book b2;
    b2.title = "C++11";
    b2.pages = 500;

    std::cout << std::boolalpha << b1.empty() << std::endl;
    b1.print();
    b2.print();
    return 0;
}
```

这个 `Book` 虽然有默认值和成员函数，但整体仍然像一个轻量数据类型：

- 数据结构简单
- 字段含义直接
- 成员函数只是对数据做自然操作
- 没有复杂封装和资源管理逻辑

这正是 `struct` 很舒服的一类使用方式。

`struct` 这一章真正有用的，不是只记住它和 `class` 的语法差别，而是学会把它当成一种意图信号：

> 这个类型主要是轻量数据载体，还是已经开始承载复杂封装规则。

读代码时，这个判断会非常省力。

----

# 推荐实践

- 纯数据模型、轻量配置对象优先考虑 `struct`。
- 需要明确封装和约束时使用 `class`。
- 简单结构体优先使用聚合初始化和成员默认值。
- 即使是 `struct`，也可以合理加入成员函数和 `const` 接口。
- 不要因为“面向对象”这几个字，就把所有简单数据类型都硬改成 `class`。
- 也不要因为 `struct` 写起来方便，就把本该封装的资源类和规则类暴露成公开字段。

----

# 小结

- 现代 `C++` 里的 `struct` 不是低配版 `class`，而是更适合轻量数据模型的一种默认表达。
- `struct` 和 `class` 的能力几乎相同，真正重要的是你想表达“公开数据”还是“封装规则”。
- 只要把“数据载体”和“强封装对象”区分开，`struct` 和 `class` 的使用边界就会清楚很多。
