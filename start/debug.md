# 调试入门

很多读者第一次学 `C++` 时，会把主要精力放在“怎样写出能编译的代码”，却没有尽早建立调试习惯。这样一来，只要程序行为不对、崩溃或者输出异常，就很容易陷入手足无措。

对 `C++11` 来说，调试不是高级技能，而是基础能力的一部分。

# 先把调试信息编进去

学习和排错阶段，建议默认带上调试信息：

```bash
g++ -std=c++11 -Wall -Wextra -pedantic -g main.cpp -o main
```

这里的 `-g` 用于生成调试符号。没有它，很多调试器就很难准确定位源码位置。

如果你在 Windows 上使用 `MSVC`，也应确认当前构建配置是调试模式，而不是只生成优化后的发布版本。

# 先学会区分 3 类问题

刚开始排错时，最有价值的不是马上背调试器命令，而是先判断问题属于哪一类：

1. 编译错误：程序还没有生成可执行文件。
2. 运行错误：程序启动后崩溃、卡住或结果异常。
3. 逻辑错误：程序不崩，但结果不对。

这三类问题的排查路线差别很大。很多初学者一看到“程序不对”就混成一团，其实先分型会清楚很多。

# 最基础的调试手段

在正式进入调试器之前，你至少要会用下面几种手段：

- 阅读编译器错误和警告
- 适量打印关键变量
- 缩小最小复现代码
- 对照输入和输出

例如：

```cpp
#include <iostream>
#include <vector>

int main() {
    std::vector<int> values{1, 2, 3};
    int sum = 0;

    for (const auto& value : values) {
        std::cout << "value = " << value << std::endl;
        sum += value;
    }

    std::cout << "sum = " << sum << std::endl;
    return 0;
}
```

打印不是最低级的办法，而是最快建立直觉的办法之一。只是当问题复杂到和生命周期、线程、栈回溯有关时，就应进入调试器。

# 断点是什么

断点可以先粗略理解成：

> 让程序在某一行暂停，好让你观察当前状态。

你通常会在这些地方打断点：

- 某个函数刚进入时
- 某个变量即将被修改前
- 某个分支判断附近
- 崩溃前最可疑的调用链位置

调试器里最常见的几个动作是：

- `step over`：执行当前行，但不进入函数内部。
- `step into`：进入当前行调用的函数内部。
- `step out`：执行到当前函数返回。
- `continue`：继续运行到下一个断点。

初学阶段只掌握这四个动作，已经足够解决很多问题。

# 栈回溯要怎么看

程序崩溃时，调试器通常会给出调用栈，也就是一串“函数是怎么一层层调用到这里的”信息。

阅读调用栈时建议：

1. 先找你自己的业务函数。
2. 再找最接近异常点的那一层。
3. 不要一开始就被大量标准库或系统库栈帧吓住。

很多时候真正有价值的问题是：

- 崩溃前最后一个业务函数是谁
- 传进来的参数是否合理
- 某个对象此时是否已经失效

# C++ 最常见的调试主题

和很多其它语言不同，`C++` 调试时有几类问题会高频出现：

- 空指针解引用
- 越界访问
- 悬空指针或悬空引用
- 对象生命周期结束后仍然使用
- 重复释放或释放后继续使用
- 线程同步错误

所以调试 `C++` 时，不能只盯着“值对不对”，还要经常问：

- 这个对象现在还活着吗
- 这个指针指向的内存还有效吗
- 这里是不是越界了
- 这个锁和线程时序是否合理

# sanitizer 很值得尽早使用

如果工具链支持，学习阶段非常推荐尝试 sanitizer。

地址相关问题可以先试：

```bash
g++ -std=c++11 -Wall -Wextra -pedantic -fsanitize=address -g main.cpp -o main
```

未定义行为相关问题可以试：

```bash
g++ -std=c++11 -Wall -Wextra -pedantic -fsanitize=undefined -g main.cpp -o main
```

这类工具经常能比肉眼更快指出：

- 越界
- 释放后使用
- 栈上对象生命周期问题
- 某些未定义行为

对初学者来说，它们的价值非常高。

# 一个典型排查例子

看下面这段代码：

```cpp
#include <iostream>
#include <vector>

int main() {
    std::vector<int> values{1, 2, 3};

    for (int i = 0; i <= values.size(); ++i) {
        std::cout << values[i] << std::endl;
    }

    return 0;
}
```

如果程序行为异常，排查思路可以先这样走：

1. 先看循环边界是否合理。
2. 观察 `i` 的取值范围。
3. 注意 `values.size()` 返回的是元素个数，不是最后一个合法下标。
4. 怀疑越界时，尽快用断点或 sanitizer 验证。

这个例子说明，调试很多时候不是“神秘技巧”，而是把代码执行路径和数据边界看清楚。

# 调试器之外也要会最小化问题

很多复杂 bug 如果直接在大工程里看，很难一下定位。更稳妥的做法通常是：

- 先删掉无关代码
- 缩成最小可复现示例
- 确认问题是否仍然存在

这样做有两个好处：

- 更容易判断是语言问题、库问题还是自己逻辑问题
- 更容易让别人帮你一起分析

这在 `C++` 里尤其重要，因为构建、模板、生命周期和链接问题一旦混在一起，噪音会非常大。

# 调试时最值得问的 6 个问题

每次排错时，你都可以快速问自己：

1. 这是编译错误、运行错误还是逻辑错误？
2. 问题第一次出现在哪个最小输入下？
3. 崩溃点附近的对象和指针是否仍然有效？
4. 是否存在越界、未初始化或生命周期结束后继续使用？
5. 是否已经打开警告、调试信息和合适的 sanitizer？
6. 我能不能把问题缩成一个更小的例子？

这套问题会比盲目反复改代码更有效。

# 与其它语言的感觉差异

如果你来自 `Go`、`PHP`、`JS`、`TS`，调试 `C++` 时最明显的区别通常是：

- 你需要更关注内存和对象生命周期。
- 有些错误不会自动抛成友好的异常。
- 程序能继续运行，不代表状态仍然安全。
- 调试器、警告和 sanitizer 的价值会更高。

也就是说，`C++` 调试往往更接近“检查程序真实状态”，而不是只看业务层输出。

# 推荐习惯

- 学习阶段默认保留 `-g`。
- 不要忽略警告。
- 怀疑内存问题时优先尝试 sanitizer。
- 先缩小问题，再深入调试器。
- 调试时优先确认生命周期、边界和所有权。

# 小结

调试 `C++11` 的核心，不是记住多少工具命令，而是尽快形成一种排查顺序：先判断问题类型，再看调用路径、变量状态、生命周期和边界，必要时借助断点与 sanitizer 验证。只要这套节奏建立起来，后面学指针、引用、类、异常和并发时，你会更有把握。
