# 函数进阶

函数在 `C++11` 中的变化非常值得重点学习。
旧时代你主要关注参数、返回值、重载、内联；到了 `C++11`，你还必须理解 `lambda`、尾置返回类型、`std::function`、右值引用参数以及可变参数模板的基本思路。

# 内联函数

```cpp
inline int square(int x) {
    return x * x;
}
```

`inline` 是一个很容易被误解的关键字。很多人第一次接触它时，会把它理解成“把函数代码直接展开到调用处”。这种理解只说对了一半。更准确地说，`inline` 有两层含义：

- 它向编译器表达“这个函数适合做内联展开”的意图。
- *它允许同一个函数定义出现在多个翻译单元中，只要这些定义完全一致。*

第二点非常重要，因为这正是很多头文件里能直接写函数定义的原因之一。

## 诞生背景

早期函数调用有固定开销，例如：

- 传参
- 跳转到函数体
- 返回调用点

如果函数非常短小，比如“平方”“取较大值”“交换两个小值”，函数调用本身的开销有时会显得比较明显。
于是编译器可以选择把函数体直接嵌入到调用处，避免一次真实调用。

> 但在现代编译器中，是否展开通常由优化器决定，而不是由你写不写 `inline` 机械决定。

```cpp
inline int addOne(int x) {
    return x + 1;
}
```

即使你写了 `inline`，编译器也可能不展开，例如：

- 函数体并不小
- 函数包含循环、复杂分支
- 递归函数不适合简单展开
- 当前编译选项没有启用相关优化

反过来，即使你没有写 `inline`，编译器也可能自动把一个非常短的函数做内联优化。

> 所以，`inline` 更适合被理解为“建议”和“链接语义”，而不是“强制命令”。

## 常见使用场景

### 头文件 inline

如果一个普通函数直接定义在头文件中，而这个头文件又被多个 `.cpp` 文件包含，链接阶段通常会报“重复定义”错误。

`inline` 可以解决这个问题：

```cpp
// math_util.h
inline int square(int x) {
    return x * x;
}
```

这样多个源文件都包含 `math_util.h` 时，只要看到的 `square()` 定义完全一致，就不会违反单一定义规则。

> 这也是为什么很多小型工具函数、模板辅助函数、头文件库中的简单函数都常配合 `inline` 使用。

### 类内定义的成员函数

在类定义内部直接写出的成员函数，通常会被视为 **隐式 `inline`** ：

```cpp
class Counter {
public:
    int value() const {
        return count_;
    }

private:
    int count_{0};
};
```

这并不意味着它一定会被编译器展开，而是说它具备 `inline` 的链接属性。

### 适合写成 inline 的函数

通常包括：

- 函数体非常短小
- 逻辑简单，几乎是一眼可懂
- 会被频繁调用
- 定义需要放在头文件中

例如简单访问器、轻量工具函数、小包装函数。

以下场景一般 **不适合** 为了“优化”而刻意写成 `inline`：

- 函数体很长
- 包含复杂控制流
- 可读性会因为放进头文件而明显下降
- 接口经常变化，放在头文件会增加编译依赖

从工程角度看，过多把实现写进头文件，往往会让编译更慢，也会让模块边界变得不清晰。

## inline 和 define

初学者有时会把 `inline` 和宏混在一起，因为它们都可能达到“像展开一样”的表面效果。但它们本质完全不同。

```cpp
#define SQUARE(x) ((x) * (x))

inline int square(int x) {
    return x * x;
}
```

`inline` 函数相对宏的优势在于：

- 有明确的参数类型
- 会参与正常的编译和类型检查
- 更容易调试
- 不容易出现重复求值等宏陷阱

例如 `SQUARE(i++)` 这种宏调用就可能产生副作用，而 `inline` 函数不会出现这种问题。

> 一句话总结：`inline` 的核心不是“手动提速”，而是“为短小函数提供更自然的头文件定义方式，并允许编译器在合适时做内联优化”。

# 尾置返回类型

当返回类型依赖复杂表达式时，`C++11` 可以使用尾置返回类型：

```cpp
auto add(int a, int b) -> int {
    return a + b;
}
```

这个特性在模板编程和 `decltype` 配合时尤其有用。它的语法形式是：

```cpp
auto 函数名(参数列表) -> 返回类型
```

之所以叫“尾置返回类型”，就是因为返回类型不再写在函数名前面，而是写到参数列表后面。

## 配合 decltype

在普通函数中，传统写法通常已经够用：

```cpp
int add(int a, int b) {
    return a + b;
}
```

- 但在 `模板代码` 里，返回类型有时依赖参数表达式，而参数名只有到函数名后面才真正出现。
- 这时如果还沿用传统写法，就会不方便甚至写不出来。
- 尾置返回类型的核心价值，就是让 `先看到参数，再描述返回类型` 成为可能。

```cpp
template <typename T, typename U>
auto add(T a, U b) -> decltype(a + b) {
    return a + b;
}
```

在普通函数里它未必比传统写法更常见，但在模板代码里往往更清晰。

`decltype(expr)` 的作用是：根据表达式 `expr` 推导出它的类型。

```cpp
int x = 0;
double y = 1.5;
decltype(x + y) z = x + y;
```

当它放在尾置返回类型中时，函数就可以把 `返回值类型等于某个表达式的结果类型` 说得非常直接。

这在泛型编程里尤其常见，因为模板参数可能是 `int`、`double`、自定义数值类型，返回类型未必固定。

## 使用建议

可以把尾置返回类型读成一句自然语言：

- `auto add(int a, int b) -> int`
- 意思是“这是一个叫 `add` 的函数，接收两个 `int`，最后返回 `int`”

当返回类型很复杂时，这种阅读顺序往往比传统写法更自然，特别是在返回类型里还要写 `decltype(...)` 时。

- 普通返回类型很简单时，传统写法通常更直接。
- 返回类型依赖表达式时，尾置返回类型更清晰。
- 在模板代码中，配合 `decltype` 使用非常常见。

# 函数指针与函数对象

在理解 `lambda` 和 `std::function` 之前，先认识两个更早出现的概念会很有帮助：

- 函数指针
- 函数对象

它们可以看作现代可调用对象体系的历史基础。

## 函数指针

函数也可以有地址，因此可以通过指针来指向一个函数。

```cpp
#include <iostream>

int add(int a, int b) {
    return a + b;
}
int main() {
    int (*op)(int, int) = add;
    std::cout << op(3, 4) << std::endl;
    return 0;
}
```

这里 `op` 就是一个函数指针，它能指向签名为 `int(int, int)` 的普通函数。

函数指针适合做一些基础回调场景，但它有明显限制：
- 只能指向普通函数
- 不能直接携带额外状态
- 语法相对难读

例如你想“既保存一个动作，又顺便记住一个局部变量”，函数指针就做不到。

## 函数对象

所谓函数对象，本质上就是“重载了 `operator()` 的对象”。

```cpp
#include <iostream>

class Adder {
public:
    explicit Adder(int base) : base_(base) {} // explicit 关键字，禁止隐式类型转换
    int operator()(int x) const {
        return base_ + x;
    }
private:
    int base_;
};
int main() {
    Adder add10(10);
    std::cout << add10(5) << std::endl;
    return 0;
}
```

这里 `add10` 看起来像函数一样可以被调用，但它其实是对象。

函数对象相对函数指针的优势在于：
- 可以保存状态
- 可以自定义行为
- 可以像普通对象一样构造、复制、传递

很多标准库算法在底层就非常适合接收这类可调用对象。

## lambda 优势

如果没有 `lambda`，很多短小逻辑其实都要额外写成函数对象类。

例如上面的 `Adder` 虽然功能不复杂，但依然需要：

- 写一个类名
- 写构造函数
- 写成员变量
- 写 `operator()`

而 `lambda` 往往可以把这些样板代码压缩成非常短的一段表达式。

> 所以你可以把 `lambda` 理解成：编译器帮你快速生成一个匿名函数对象。

## 差异对比

可以用下面这条线来理解它们：

- 普通函数：最直接的命名行为
- 函数指针：让普通函数可以像数据一样被传递
- 函数对象：让“可调用行为”也能携带状态
- `lambda`：让函数对象的写法变得非常轻便
- `std::function`：把各种可调用对象统一装进一个接口里

这条演进线背后，其实是在解决同一个问题：**如何把“行为”像“数据”一样传递、保存和组合**

### 直观对比

- 如果目标只是“传入一个不依赖外部状态的普通规则”，普通函数和函数指针就可能够用了。
- 如果目标是“在调用时顺便记住一些上下文”，函数对象和 `lambda` 会更自然。
- 如果目标是“对外只暴露统一签名，不关心内部到底是什么可调用对象”，那就轮到 `std::function` 出场。

# lambda 表达式

这是 `C++11` 函数相关最重要的新特性之一。

```cpp
#include <iostream>

int main() {
    auto add = [](int a, int b) {
        return a + b;
    };
    std::cout << add(2, 5) << std::endl;
    return 0;
}
```

`lambda` 的价值在于：

- 可以就地定义短小逻辑
- 非常适合与算法库配合
- 避免为了简单回调额外定义函数对象类

## 本质

- 表面上看，`lambda` 很像“没有名字的函数”。但从语言本质上说，它更接近 `编译器自动生成的函数对象`。
- 你可以把它粗略理解成编译器帮你生成了一个带有 `operator()` 的对象，只是语法更短、更方便。
- 这也是为什么 `lambda` 可以携带捕获状态，而普通函数不行。

它的基本形式可以概括为：

```txt
[捕获列表](参数列表) -> 返回类型 {
    // 函数体
}
```

各部分含义分别是：

- `[捕获列表]`：决定是否把外部变量带进 `lambda`
- `(参数列表)`：和普通函数参数列表类似
- `-> 返回类型`：**可选**，必要时显式写出
- `{}`：函数体

最简单的 `lambda` 甚至可以什么都不捕获：

```cpp
auto hello = []() {
    std::cout << "hello" << std::endl;
};
```

### 现代C++中的重要性

现代 `C++` 很强调：

- 把逻辑放到最接近使用点的位置
- 用算法表达“做什么”，而不是手写底层循环
- 把回调、条件、规则写成可调用对象传来传去

`lambda` 正好非常适合这些场景。

比如配合算法库：

```cpp
#include <algorithm>
#include <vector>
#include <iostream>

int main() {
  std::vector<int> nums{1, 2, 3, 4, 5};
  auto count = std::count_if(nums.begin(), nums.end(), [](int x) {
     return x % 2 == 0;
  });
  std::cout << "count: " << count << std::endl;
  return 0;
}
```

如果没有 `lambda`，你往往需要额外定义一个独立函数或函数对象类，代码会更分散。

## 捕获列表

常见捕获形式：

- `[x]` 按值捕获
- `[&x]` 按引用捕获
- `[=]` 按值捕获所有外部变量
- `[&]` 按引用捕获所有外部变量

注意点：
- 按引用捕获很方便，但要小心生命周期问题。
- 这里最关键的理解是：捕获决定了 `lambda` 如何访问外部作用域里的局部变量。
- 普通函数无法直接“记住”调用点附近的局部状态，而 `lambda` 可以通过捕获把这些状态带进来。

### 值捕获、引用捕获

```cpp
int base = 10;
auto f1 = [base](int x) {
    return base + x;
};
auto f2 = [&base](int x) {
    return base + x;
};
```

- `[base]`：**值捕获**，把当前 `base` 的值复制一份进 `lambda`，**默认只读**，外部 `base` 后续变化，未必影响按值捕获的结果
- `[&base]`：**引用捕获**，不复制，而是直接引用外部那个 `base`，外部对象如果已经销毁，按引用捕获就会有风险

### 默认捕获

`[=]` 和 `[&]` 很方便，但在代码变长后，读者不容易一眼看出到底捕获了哪些变量。

所以在教学和工程实践里，一个常见建议是：

- 局部逻辑很短时，可以使用默认捕获
- 逻辑稍复杂时，显式写出需要捕获的变量通常更清晰

**按值捕获的变量默认是只读的**，可以用 `mutable` 让 `lambda` 修改它自己的那份副本：

```cpp
int x = 10;
auto f = [x]() mutable {
    ++x;  // 如果不加 mutable 编译报错，increment of read-only variable 'x'
    return x;
};
```

这里修改的是捕获进来的副本，不是外部原变量。

### 生命周期

*按引用捕获最大的风险，就是 `lambda` 被保存下来后，外部变量却已经离开作用域。*
例如下面这种思路就是危险的：

```cpp
// 伪示例：不要这样返回引用捕获外部局部变量的 lambda
// auto makeFunc() {
//     int x = 10;
//     return [&x]() { return x; };
// }
// x 在函数返回后就已经销毁，而返回出去的 `lambda` 还试图引用它，这会导致未定义行为。
```

> 所以只要 `lambda` 可能被延迟执行、跨作用域保存、传给异步任务，就要特别小心引用捕获。

## 作为参数

`lambda` 最常见的用途之一，就是把它当作“行为参数”传给算法或其他函数。

```cpp
#include <iostream>

template <typename Func>
void repeat(int n, Func func) {
    for (int i = 0; i < n; ++i) {
        func("haha", i);
    }
}
int main() {
    repeat(3, [](std::string name, int i) {
        std::cout << name << " " << i << std::endl;
    });
    return 0;
}
```

> 这里函数接收的不是某个固定类型的“普通函数”，而是一个可调用对象。现代 `C++` 中大量接口都会这样设计。

## 作为返回值

除了把 `lambda` 作为参数传进去，现代 `C++` 也经常把“一个行为”作为返回值返回出来。

```cpp
#include <iostream>

auto makeAdder(int base) {
    return [base](int x) {
        return base + x;
    };
}
int main() {
    auto add10 = makeAdder(10);
    std::cout << add10(5) << std::endl;
    return 0;
}
```

这里 `makeAdder()` 返回的不是一个普通整数，而是一个“记住了 `base` 的可调用对象”。

这类写法很重要，因为它体现了现代 `C++` 一个典型思想：

- 数据可以作为参数传递
- **行为也可以作为参数或返回值传递**

> 从这个角度看，函数不再只是“执行代码的地方”，它也可以是“生产行为的工厂”。

返回 `lambda` 适合下面这些思路：

- 根据不同配置，构造不同的处理规则
- 先绑定一部分参数，再把剩余逻辑交给后续调用
- 让接口返回一个“将来再执行”的动作

这和函数对象、回调机制、策略模式之间都有很强的联系。

## 适用场景

- 逻辑短小，且只在当前位置使用一次
- 需要传给算法、回调、事件处理器
- 逻辑依赖周围局部变量，使用捕获会更自然

如果逻辑已经很长、需要复用、需要单独命名表达语义，那就更适合写成普通函数或函数对象。

# std::function

```cpp
#include <functional>
#include <iostream>

int main() {
    std::function<int(int, int)> op = [](int a, int b) {
        return a + b;
    };
    std::cout << op(3, 7) << std::endl;
    return 0;
}
```

当你需要“存储一个可调用对象”时，`std::function` 非常实用，它能统一包装：

- 普通函数
- `lambda`
- 函数对象
- `bind` 结果

```cpp
int add(int a, int b) {
    return a + b;
}
std::function<int(int, int)> op1 = add;
std::function<int(int, int)> op2 = [](int a, int b) {
    return a - b;
};
```

- `std::function<R(Args...)>` 更像一个“可调用对象容器”，只要某个对象能以指定签名被调用，它就能装进去。
- 相比直接使用模板参数或直接调用具体 `lambda`，`std::function` 往往有额外包装开销。因此：
  - 需要统一接口、存储回调时很适合用它
  - 对性能特别敏感的底层代码中，要留意是否真的需要它

再简单区分一下：

- `lambda`：用来“创建一个可调用对象”
- `std::function`：用来“保存、传递一个统一签名的可调用对象”

两者经常一起出现，但它们解决的问题并不相同。

## 选择函数类型

这是现代 `C++` 里非常常见的一个实际问题。

### 普通函数

- 逻辑稳定、明确，值得拥有一个清晰名字
- 不依赖外部局部状态，不需要捕获
- 需要在多个地方复用

> 普通函数的优点是语义清楚、可复用、可测试，也最容易被读者快速识别。

### lambda

- 逻辑很短，只在当前位置使用
- 作为算法、回调、过滤条件传入
- 需要捕获当前作用域中的局部变量

> `lambda` 的最大优势是“离使用点近”，读代码时上下文很完整。

### std::function

- 你需要把可调用对象保存到变量、成员或容器中
- 你希望接口统一接收某一种调用签名
- 你不关心具体传进来的是普通函数、`lambda` 还是函数对象

```cpp
#include <functional>

class Button {
public:
    void setOnClick(std::function<void()> handler) {
        onClick_ = handler;
    }

    void click() {
        if (onClick_) {
            onClick_();
        }
    }

private:
    std::function<void()> onClick_;
};
```

这里 `Button` 不关心点击事件最终来自哪里，它只要求对方“能像 `void()` 这样被调用”。

### 一个简单判断思路

可以先这样选：

- 想表达一个稳定、可复用的命名逻辑，优先普通函数
- 想就地写一个短小行为，优先 `lambda`
- **想统一保存和传递各种可调用对象，再考虑 `std::function`**

这样理解后，这三者就不是彼此竞争关系，而是处在不同层次的工具：

- 普通函数：命名好的固定行为
- `lambda`：就地创建的局部行为
- `std::function`：统一封装行为的容器

# 右值引用参数

`C++11` 引入右值引用后，函数参数的设计也升级了：

```cpp
#include <iostream>
#include <string>

void setName(std::string&& name) {
    std::cout << name << std::endl;
}

int main() {
    setName("temporary");
    return 0;
}
```

这和移动语义密切相关，后续在引用和类设计章节会重点展开。

一个更常见的理解方式是：右值引用常用于“接收即将被销毁的临时对象”，从而把它内部的资源移动过来，而不是再复制一份。

```cpp
void setName(const std::string& name) {
    std::cout << "copy or read: " << name << std::endl;
}

void setName(std::string&& name) {
    std::cout << "move-ready: " << name << std::endl;
}
```

这类接口设计在标准库容器和现代类设计中非常常见。

# 可变参数模板

`C++11` 支持模板参数包，这是很多现代库设计的基础：

```cpp
#include <iostream>

void print() {
    std::cout << std::endl;
}

template <typename T, typename... Args>
void print(const T& first, const Args&... rest) {
    std::cout << first << " ";
    print(rest...);
}

int main() {
    print(1, "hello", 3.14);
    return 0;
}
```

它的核心价值不只是“参数个数可变”，更重要的是可以在编译期保持类型信息。
这比传统 `printf` 式可变参数更安全，也更符合 `C++` 的类型系统。

# 推荐实践

- 短小回调逻辑优先使用 `lambda`。
- `inline` 主要用于短小函数和头文件定义，不要把它当成万能性能开关。
- 只有当返回类型不清晰或依赖表达式时，再使用尾置返回类型。
- 逻辑需要命名和复用时优先普通函数，局部短逻辑再考虑 `lambda`。
- 只需要指向普通函数时可以考虑函数指针；需要携带状态时优先函数对象或 `lambda`。
- 需要统一保存回调时使用 `std::function`。
- 捕获引用时始终检查被捕获对象的生命周期。

# 小结

- 函数在现代 `C++` 里不只是“放逻辑的地方”，它也是接口设计的核心位置。
- 一个函数如何声明、参数如何传递、是否适合写成 `inline`、回调是否用 `lambda`，这些都在表达设计意图。

如果你能从“语法记忆”进一步进入“接口语义”的角度去理解函数，那么后面学习类、模板、STL 算法和泛型编程时，会轻松很多。
