# 引用

引用在旧版 `C++` 中就已经是核心特性，但到了 `C++11`，它的作用被彻底扩展了。  
过去你可能主要把引用理解成“别名”或“避免复制”；现在你还必须掌握右值引用 `T&&`，因为它直接关系到移动语义、性能优化和现代类设计。

这也是为什么很多人学到这一章时会突然卡住。难点通常不是左值引用本身，而是当你同时碰到：

- 左值与右值
- 左值引用
- 常量引用
- 右值引用
- `std::move`
- 拷贝与移动
- 转发引用

这些概念时，很容易产生一种感觉：每个词都认识，但放到一起就开始模糊。  
更好的做法不是一上来硬记定义，而是先建立一个够用的模型。  
先记住一句最重要的话：

> 引用最核心的意义，不是少写一个 `*`，而是表达“这个对象是 `借来用的`，还是可以 `被移动的`”。

和上一章《指针》相比，这一章更关注三件事：

- 借用语义
- 值类别
- 移动语义

如果说指针一章更强调“地址、可空性、所有权边界”，那么这一章更强调“别名、借用和转移”。

----

# 核心作用

很多初学者第一次看到引用，会觉得它和指针很像，只是写法更顺手。这种理解不算错，但还不够。  
在现代 `C++` 里，引用至少承担着三类职责：

- 给已有对象起别名
- 作为函数参数，表达“借用而非拥有”
- 配合右值语义，表达“这个对象可以被移动”

> 所以引用不是一个边缘语法点，而是接口设计里非常高频的一部分。

例如下面这类代码，你在工程里会经常看到：

```cpp
void print(const std::string& text);
void setName(std::string&& name);
```

这两个参数虽然都写成“引用”，但表达的语义完全不同：

- `const std::string&`：只读借用
- `std::string&&`：准备接收并转移一个临时资源

也正因为如此，学引用时不能只盯着 `&` 和 `&&` 的符号本身，而要一直问：

- 这个接口是不是在借对象
- 借的时候能不能修改
- 这个对象是否可能被移动

----

# 值类别

理解 `C++11` 引用前，必须先区分左值和右值。

- **左值：** 有持久身份、通常能稳定取地址、后续还能继续使用的对象
- **右值：** 临时值、字面量、即将销毁的值，往往更适合被移动资源

```cpp
int a = 10; // a 是左值
10;         // 10 是右值
```

对入门阶段来说，可以先建立一个够用的第一模型：
> - `左值：` 有名字、能稳定取地址、通常还能继续使用
> - `右值：` 临时值、将要结束生命周期、可以安全转移资源

然后把不同引用形式一起理解：
> - `T&`：给稳定对象起别名
> - `const T&`：只读借用
> - `T&&`：接住一个可被移动的对象

这个模型不覆盖全部语言细节，但已经足够支撑大量工程代码阅读。  
先把这套直觉建立起来，再去看更复杂的模板规则，会轻松很多。

一个更直观的例子：

```cpp
#include <string>

std::string name = "cpp";
std::string makeName() {
    return "temp";
}
```

这里通常可以这样理解：

- `name` 是左值，因为它有稳定身份
- `makeName()` 的返回结果通常是右值，因为它是一个临时结果
- 字面量和临时构造结果也通常按右值理解

这套直觉虽然不覆盖全部标准细节，但足够你先把大部分代码读顺。

# move 语义

最容易被误解的点是：`std::move` 这个名字看起来像“执行移动”，但它本身并不真的移动任何资源。
> `std::move` 一个贴标签机。把还要用的“左值”伪装成快死掉的“右值”，告诉编译器：“我不要它了，你可以移走它的资源”。


```cpp
#include <iostream>
#include <string>
#include <utility>

int main() {
    std::string a = "hello world";
    std::string b = std::move(a);

    std::cout << "b = " << b << std::endl;
    std::cout << "a = " << a << std::endl;
    return 0;
}
// 输出：
// b = hello world
// a =
```

*真正是否发生转移，要看目标类型是否提供了合适的移动构造、移动赋值或其他移动操作。*

所以很多初学者会`误以为`：写了 `std::move`，对象就一定已经被搬空了。  
这个理解并不准确。正确理解应该是：
> - `std::move` 只是发出“你可以把它当右值处理”的信号
> - 后续代码是否真的移动资源，要看类型实现

一个非常实用、也非常容易考到的结论是：
> - 被 `std::move` 过的对象仍然有效
> - 但它的值通常处于未指定状态

这意味着你不能再假设它还保留原来的内容，但它依然可以被：
> - 重新赋值
> - 销毁
> - 执行那些对“有效但值未指定”仍然安全的操作

所以使用 `std::move` 前，最好先问自己一句：**我是不是已经准备好放弃这个对象当前的值了？**

# 左值引用

最基本的引用写法：

```cpp
#include <iostream>

int main() {
    int value = 10;
    int& ref = value;

    ref = 20;
    std::cout << value << std::endl;
    return 0;
}
```

**引用本质上是对象的别名，不是重新拷贝一份。** 所以在这个例子里：

- `ref` 和 `value` 表示的是同一个对象
- 改 `ref`，就是在改 `value`
- 它们的地址相同
- 引用一旦绑定，通常就不会再改成引用别的对象

你可以先把 `T&` 理解成一句很实用的话：**给一个稳定存在的对象起别名。**

## 基本规则

左值引用有几个非常重要的基本规则：

- 定义时通常必须初始化
- 不能为 `null`
- 绑定之后不能改绑到另一个对象
- `它本身不是一个新对象，而是原对象的另一个名字`

例如下面这些写法就很有代表性：

```cpp
int a = 10;
int b = 20;

int& ref = a;
ref = b;      // 这不是让 ref 改绑到 b，而是把 b 的值赋给 a
```

执行后：

- `a` 会变成 `20`
- `ref` 仍然绑定 `a`

这也是引用最容易被误解的地方之一。

## 常见用途

最常见的用途有三个：

- 给变量取别名
- 作为可修改参数传给函数
- 作为返回值，直接操作原对象

```cpp
#include <iostream>

void increase(int& value) {
    value++;
}

int main() {
    int number = 5;
    increase(number);
    std::cout << number << std::endl;
    return 0;
}
```

这里 `increase` 没有返回值，但它依然修改了调用方的变量，因为参数 `value` 是 `number` 的别名。

# 右值引用

`C++11` 引入了右值引用：

```cpp
int&& r = 10;
```

它的意义不是“多了一种更快的引用”，而是让语言终于能区分：
- 这个对象是长期存在、还要继续使用的
- 还是一个临时对象，可以安全地转移内部资源

这正是现代 `C++` 能高效支持移动语义的关键。
所以更准确地说，右值引用表达的是一种语义：
> 我接到的是一个可以被移动、可以把内部资源转交出去的对象。

这个语义和性能高度相关，但它本身不是“性能黑魔法”，而是对象生命周期信息的一部分。  
和 `左值引用` 最本质的差别不是写法，而是绑定对象的类别：

- `T&` 主要绑定左值
- **`const T&` 能绑定左值，也能绑定右值**
- `T&&` 主要绑定右值

```cpp
int a = 10;

int& r1 = a;
const int& r2 = 20;
int&& r3 = 30;
```

这些写法都合法，但：

- `r1` 绑定的是可长期存在的对象
- `r2` 是只读引用，也可以绑定临时值
- `r3` 明确接住了一个右值

## 配合 move

`右值引用` 与 `std::move` 必须配合使用吗？
> 不是必须配合使用，但它们是“天生一对”。​

### 临时对象

如果​本来就是 `纯临时对象（右值）`​​，自己会销毁，不需要贴标签（不需要 `std::move`）

```cpp
void process(std::string&& str) { // 这是一个废品回收站
    // 移走 str 的资源
}

// 调用时：
process(std::string("我是临时造出来的字符串")); 
// std::string("...") 本来就是个临时对象（右值），用完就没。
// 编译器非常聪明，自动把它送进 && 接收器，这里完全不需要写 std::move。
```

### 只写 move

```cpp
void process(const std::string& str) { // 普通复印店，不偷资源，只拷贝
    std::string copy_str = str;
}

std::string a = "我的自行车";
process(std::move(a)); // 你给 a 贴了 可移动 标签，送了过去
```

程序不会报错，但​​移动不会发生，依然会执行老老实实的“拷贝”​​。  
因为接收方没有 && （右值引用），看不懂标签，只能按老规矩复印一份。  
变量 a 依然完好无损。

### 转移左值

一个​​有名字的变量（左值）​​，​​明确想把它的资源转移给别人​​时，二者就必须配合了。 
必须用 `std::move` 贴标签，并且接收方必须有 `&&` 准备好接收。

```cpp
std::string a = "很长的数据";
std::string b = std::move(a); 
// 1. std::move(a) 给 a 贴上标签。
// 2. std::string b 的底层有一个“移动构造函数”（参数是 &&）。
// 3. b 看到 a 的标签，立刻把 a 的资源偷过来。
```

## 工程价值

右值引用和移动语义出现后，`C++11` 能在很多场景减少不必要复制：

- 函数返回大对象
- 容器扩容搬迁元素
- 接口接收临时资源
- 类的移动构造和移动赋值

这也是为什么这一章不只是“语法扩展”，它其实和性能、接口设计都关系很大。

看一个更接近项目代码的例子：

```cpp
#include <iostream>
#include <string>
#include <utility>

class User {
public:
    explicit User(std::string name)
        : name_(std::move(name)) {
    }
    void print() const {
        std::cout << name_ << std::endl;
    }

private:
    std::string name_;
};

int main() {
    std::string name1 = "name1";
    // name1 复制了一份传递给构造函数，后续的事情与 name1 无关
    // 所以 name1 保持不变
    User user1(name1);
    user1.print();
    std::cout << "name1 = " << name1 << std::endl;

    std::string name2 = "name2";
    // std::move 给 name2 贴上了 `可转移` 标签
    // 于是编译器不复制，直接把 name2 的资源传递给 构造函数
    // 构造函数内部 又把 name2 的资源传递给了 user2.name_
    // 所以 name2 被清空了
    User user2(std::move(name2));
    user2.print();
    std::cout << "name2 = " << name2 << std::endl;
    return 0;
}
```

程序输出：

```txt
name1
name1 = name1
name2
name2 =
```

# 引用与指针

这两个概念很容易放在一起学，因为它们都能“间接访问对象”，但语义其实不同。

先看最直观的对比：

```cpp
int value = 10;

int* ptr = &value;
int& ref = value;
```

它们都能访问 `value`，但差别至少有这些：

- 指针可以为空，引用通常不能为空
- 指针可以改指向别的对象，引用绑定后通常不能改绑
- 指针访问成员常写 `*`、`->`，引用使用时更像对象本身
- 指针更适合表达 `“可能没有对象”或“需要手动处理地址”`
- 引用更适合表达 `“这里一定有一个对象，我只是借来用”`

可以把它们先粗略理解成：

- **指针：** 更底层、更灵活，也更容易出错，适用场景：
  - 参数允许“不传对象”
  - 需要和 C API、底层接口、裸地址打交道
  - 需要显式表达“可为空”
  - 需要做数组、内存块、地址运算这类底层操作
- **引用：** 更高层、更偏接口语义表达，使用场景：
  - 参数一定非空
  - 只是借用对象，不涉及所有权
  - 不想让调用方看到“地址操作”的细节
  - 希望接口更自然，像直接操作对象

所以现代 `C++` 的一个常见判断习惯是：
> - 一定存在的对象，用引用
> - 可能不存在的对象，用指针

# 常量引用

工程里最常见的引用形式之一，是 `const T&`。
如果你刚从前面的 `常量` 一章过来，可以把那里当成“建立只读边界概念”，把这里当成“把只读借用真正放进参数设计和接口语义里”。

```cpp
#include <iostream>
#include <string>

void print(const std::string& text) {
    std::cout << text << std::endl;
}
```

它非常常见，是因为它同时具备几个重要优点：

- 避免复制大对象
- 保证只读
- 可以绑定临时对象

这使它特别适合做“只读输入参数”。  
很多标准库接口、很多项目代码里的函数参数，都会优先用 `const T&` 表达“我只借来看，不修改，也不接管所有权”。

----

# 返回引用

引用不仅能做参数，也能做返回值。但这里有一个非常重要的前提：
> 返回的引用必须指向一个在函数返回后仍然活着的对象。

```cpp
#include <string>

// 这里返回的是调用方传进来的对象引用，所以通常是安全的。
std::string& good(std::string& a, std::string& b) {
    return a;
}

// `text` 在函数结束时就销毁了，返回出去的引用会变成悬空引用。
std::string& bad() {
    std::string text = "hello";
    return text;  // 错误：返回了局部变量的引用
}
```

**悬空引用** 和 **悬空指针** 本质上是同一类问题：  
你手里还拿着一个“看起来能访问对象”的东西，但对象本身已经不存在了。  
这类 bug 的危险在于：

- 有时立即崩溃
- 有时只是输出奇怪结果
- 有时在测试里正常，线上才出问题

因此，返回引用时一定先问：
> - 它引用的对象是谁
> - 那个对象在函数返回后还活着吗

# 参数传递

这是引用最常见、也最容易写乱的地方。  
很多设计问题，本质上都在问：这个函数到底是借对象、修改对象，还是接管对象。
可以先建立一套简单的经验规则。

## 只读参数

如果对象比较大，而且函数只读不改，优先考虑：

```cpp
void print(const std::string& text);
```

适用场景：

- `std::string`
- `std::vector`
- 自定义大对象

## 修改参数

如果函数需要直接改外部对象，可以考虑：

```cpp
void sort(std::vector<int>& values);
```

这里 `values` 是调用方对象的别名，所以函数内部修改会反映到外面。

## 可空参数

如果参数允许“不传对象”，很多时候更适合用指针：

```cpp
void print(const std::string* text);
```

因为引用通常表达“这里一定有对象”。  
也正因为如此，`const T&` 和 `const T*` 虽然都能表达只读，但前者更强调“必定存在”，后者更强调“允许为空”。

## 转移资源

如果函数要接住一个临时值、并准备移动其内容，可以考虑：

```cpp
void setName(std::string&& name);
```

不过在很多现代类设计里，还有一种非常常见的写法：

```cpp
class User {
public:
    explicit User(std::string name)
        : name_(std::move(name)) {
    }

private:
    std::string name_;
};
```

这个构造函数很典型：
- 参数按值接收
- 进入对象内部后再 `std::move`

这种写法在现代 `C++` 里很常见，因为它能同时兼顾：

- 调用时的便利性
- 左值实参和右值实参的统一入口
- 进入成员后再利用移动语义减少额外复制

因此，参数设计不要只死记某一个规则，而要结合：

- 类型大小
- 是否需要修改
- 是否允许空值
- 是否准备接管资源

# 转发引用

这是更进阶的话题，但你至少要知道下面这种写法在模板里很重要：

```cpp
template <typename T>
void f(T&& value) {
}
```

在模板推导语境中，这里的 `T&&` 不一定总是普通的右值引用。它可能形成所谓的“转发引用”或“万能引用”。

为什么这件事重要？因为现代泛型代码经常希望做到：

- 传进来的是左值，就按左值语义处理
- 传进来的是右值，就按右值语义处理

而转发引用正是实现这种“保留值类别信息”的关键基础之一。

入门阶段不用急着把全部规则背下来，但你至少要先知道：

- 不是所有 `T&&` 都表示“纯右值引用”
- 模板推导一参与进来，语义就可能发生变化
- 它通常会和 `std::forward` 一起出现

例如：

```cpp
#include <utility>

template <typename T>
void wrapper(T&& value) {
    process(std::forward<T>(value));
}
```

这里 `std::forward` 的作用，不是简单移动，而是尽量保留传进来时的左值/右值属性。

你现在不必把规则记到很细，但至少要知道：

- `std::move` 更像“强制按右值看待”
- `std::forward` 更像“按原来的值类别转发出去”

# 综合示例

```cpp
#include <iostream>
#include <string>
#include <utility>

void print(const std::string& text) {
    std::cout << "const ref: " << text << std::endl;
}

void print(std::string&& text) {
    std::cout << "right value ref: " << text << std::endl;
}

void printMove(std::string text) {
    std::string tmp = std::move(text);
    std::cout << "moved value: " << tmp << std::endl;
}

int main() {
    print("temporary");
    
    std::string name = "cpp11";

    print(name);
    std::cout << "name1 = " << name << std::endl;

    print(std::move(name));
    std::cout << "name2 = " << name << std::endl;

    printMove(name);
    std::cout << "name3 = " << name << std::endl;
    
    printMove(std::move(name));
    std::cout << "name4 = " << name << std::endl;
    return 0;
}

```

这个例子很适合用来把前面的概念串起来：

- `print(name)` 传入左值，匹配 `const std::string&`
- `print("temporary")` 传入临时值，匹配 `std::string&&`
- `print(std::move(name))` 把左值显式转换成右值表达式，也会匹配 `std::string&&`

这说明同一个接口可以根据左值/右值自动选择不同重载，而这正是现代 `C++` 能同时兼顾语义清晰和性能优化的重要基础。

# 常见误区

- 误把引用当成“更安全的指针”。引用确实也能间接访问对象，但它更重要的价值在于表达借用语义。
- 误把右值引用理解成“更快的引用”。右值引用首先是在表达“这个对象可以被移动”。
- 误以为写了 `std::move` 就一定更高效。是否真的发生高效转移，取决于类型是否提供了合理的移动实现。
- 误以为被移动后的对象已经彻底失效。它仍然有效，只是值通常不该再做假设。
- 误以为返回引用总比返回值高效。返回引用如果生命周期不对，风险非常大；而现代编译器对返回值优化已经很成熟。
- 误以为学引用只要记住 `&` 和 `&&` 的语法。真正关键的是理解左值、右值、借用、移动和生命周期之间的关系。

----

# 推荐实践

- 先把左值/右值的直觉建立起来，再深入模板转发细节。
- 只读大对象参数优先使用 `const T&`。
- 需要修改调用方对象时，再使用 `T&`。
- 参数可能为空时，优先考虑指针而不是引用。
- 需要接收并转移临时资源时，再考虑 `T&&`。
- 只有确实准备把对象当右值处理时，才调用 `std::move`。
- 被 `std::move` 过的对象仍然有效，但值处于未指定状态，不要假定其内容。
- 返回引用前一定确认被引用对象在函数返回后仍然存活。
- 读类设计时，特别关注其是否定义了移动构造和移动赋值。
