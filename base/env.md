# 环境准备

开始重新学习 `C++` 时，第一件事不是背语法，而是确保你的编译环境真的在用 `C++11`。很多“代码为什么不能编译”的问题，本质上只是编译器标准版本不对。

# 推荐编译器

常见选择：

- `g++`，适合 Linux、WSL、MinGW 环境
- `clang++`，错误信息友好，诊断体验通常更好
- `MSVC`，Windows 项目中非常常见

如果你要接手现有项目，优先确认：

- 项目使用哪套编译器
- 是否显式开启了 `C++11`
- 构建系统是 `Makefile`、`CMake` 还是 IDE 工程

# 启用 C++11

对于 `g++` 或 `clang++`，最常见的命令是：

```bash
g++ -std=c++11 main.cpp -o main
```

如果不写 `-std=c++11`，编译器可能会：

- 默认启用较老标准
- 或启用更高标准，但行为和项目配置不一致

接手项目时，最好和项目配置保持一致，不要一开始就私自升级到 `C++17` 或 `C++20`。

# 验证程序

用下面这段代码验证你的环境是否支持 `C++11`：

```cpp
#include <iostream>
#include <vector>

int main() {
    std::vector<int> data{10, 20, 30};

    for (auto item : data) {
        std::cout << item << std::endl;
    }

    return 0;
}
```

如果下面这些语法都能编译通过，基本说明 `C++11` 已经正常启用：

- 列表初始化 `{10, 20, 30}`
- `auto`
- 范围 `for`

# 目录结构

学习阶段建议按“一个知识点一个文件”的方式组织示例：

```text
demo/
  auto_demo.cpp
  lambda_demo.cpp
  move_demo.cpp
```

这样做的好处是：

- 每个示例都能独立编译
- 容易定位错误
- 便于对比不同语法点

# 常用编译参数

除了 `-std=c++11`，还推荐在学习阶段开启更多警告：

```bash
g++ -std=c++11 -Wall -Wextra -pedantic main.cpp -o main
```

这些参数有助于你尽早发现：

- 隐式类型转换问题
- 未使用变量
- 可疑的控制流
- 旧式写法带来的潜在风险

# Makefile 示例

如果你希望多次编译更方便，可以准备一个最小 `Makefile`：

```makefile
CXX = g++
CXXFLAGS = -std=c++11 -Wall -Wextra -pedantic

all: demo

demo: main.cpp
	$(CXX) $(CXXFLAGS) main.cpp -o demo
```

# CMake 配置

很多现代 `C++` 项目都使用 `CMake`。最常见的写法如下：

```cmake
cmake_minimum_required(VERSION 3.5)
project(cpp11_demo)

set(CMAKE_CXX_STANDARD 11)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

add_executable(cpp11_demo main.cpp)
```

# 版本检查

你可以用宏 `__cplusplus` 粗略判断编译标准：

```cpp
#include <iostream>

int main() {
    std::cout << __cplusplus << std::endl;
    return 0;
}
```

在支持 `C++11` 的环境里，这个值通常至少是 `201103L`。

# 学习建议

- 先使用命令行编译，理解“源文件 -> 编译 -> 链接”的过程。
- 每个示例都亲手编译一次，不要只看不跑。
- 一旦示例涉及资源管理，优先尝试智能指针和标准库容器。
- 养成看编译警告的习惯，不要只盯着 error。

# 小结

环境准备的核心不是“装好编译器”这么简单，而是明确你正在使用的标准、编译参数和构建方式。只有环境稳定了，后续学习 `auto`、`lambda`、移动语义这些 `C++11` 特性时，结论才不会被工具链噪音干扰。

## 其它语言开发者最容易忽略的部分

如果你来自 `Go`、`PHP`、`JS`、`TS`，一开始最容易把“环境准备”理解成：

- 装一个编译器
- 写个 hello world
- 能跑就行

但在 `C++` 里，这样通常还不够。因为后面你会不断遇到下面这些问题：

- 某个头文件找不到，到底是路径问题还是构建系统问题
- 某段代码能编译，但链接时报未定义引用
- 项目明明能在别人电脑上编译，自己这里却因为标准版本不同而失败
- 有些 bug 不是逻辑错，而是因为没有打开足够警告或调试信息

所以这章更现实的目标是：把“能跑起来”升级成“能稳定复现实验、能读懂错误、能和项目配置保持一致”。

## 编译错误和链接错误要分清

这是 `C++` 入门时极其重要的一点。

### 编译错误

通常出现在单个源文件编译阶段，例如：

- 语法写错
- 类型不匹配
- 头文件缺失
- 名字未声明

### 链接错误

通常出现在多个目标文件合并阶段，例如：

- 函数声明了但没有定义
- 对应 `.cpp` 没有被编译进来
- 同名符号被重复定义

很多初学者一看到红字就慌，但只要先判断它属于“编译”还是“链接”，排查路径会清晰很多。

## 一个最小多文件实验

建议你不仅编译单文件示例，也自己做一个最小多文件实验：

`main.cpp`

```cpp
#include <iostream>
#include "math.h"

int main() {
    std::cout << add(3, 4) << std::endl;
    return 0;
}
```

`math.h`

```cpp
#ifndef MATH_H
#define MATH_H

int add(int a, int b);

#endif
```

`math.cpp`

```cpp
#include "math.h"

int add(int a, int b) {
    return a + b;
}
```

然后执行：

```bash
g++ -std=c++11 -Wall -Wextra -pedantic main.cpp math.cpp -o demo
```

如果你把 `math.cpp` 从命令里删掉，就能直观看到链接错误长什么样。这种实验会比纯阅读更容易帮你理解构建模型。

## 学习阶段推荐保留调试信息

很多人只关心“能不能出可执行文件”，但学习和排错阶段更建议保留调试信息：

```bash
g++ -std=c++11 -Wall -Wextra -pedantic -g main.cpp -o main
```

`-g` 的作用是生成调试符号，方便你使用调试器定位问题。即使你暂时不深入调试器，也建议养成这个习惯。

## sanitizer 非常值得尽早打开

对初学者来说，`C++` 的很多坑来自越界、悬空引用、释放后使用等问题。工具链支持时，建议尽早体验 sanitizer：

```bash
g++ -std=c++11 -Wall -Wextra -pedantic -fsanitize=address -g main.cpp -o main
```

以及：

```bash
g++ -std=c++11 -Wall -Wextra -pedantic -fsanitize=undefined -g main.cpp -o main
```

它们能帮你更早发现“程序虽然能跑，但已经很危险”的问题。

## Windows、Linux、WSL 的差异该怎么面对

如果你在 Windows 上学习 `C++`，很可能会同时看到：

- `MSVC`
- MinGW 的 `g++`
- WSL 里的 `g++`
- `clang++`

初学阶段最重要的建议是：

- 先固定一套环境
- 先固定一个标准版本
- 先固定一组编译参数

不要在还没建立基础直觉时频繁切换工具链，否则很多问题会混在一起，难以判断到底是语言问题、平台问题，还是构建配置问题。

## 命令行编译值得先学会

即使以后你主要使用 IDE，也建议先熟悉最小命令行编译流程。原因是：

- 能帮你看清源文件、编译参数和输出结果之间的关系
- 出问题时更容易排除 IDE 配置噪音
- 有助于理解 `Makefile`、`CMake` 最终到底在做什么

你可以把 IDE 看成“帮你自动生成和执行很多编译命令的工具”，而不是神秘黑盒。

## 一个推荐的学习目录

除了前面的一文件一知识点，还建议你逐步升级为：

```text
demo/
  01_lang/
    main.cpp
  02_function/
    main.cpp
  03_header_build/
    main.cpp
    math.cpp
    math.h
  04_container/
    main.cpp
```

这样做有几个好处：

- 每个主题可以独立验证
- 容易回头复习
- 多文件案例和单文件案例可以并存

## 学习阶段至少养成这几个习惯

- 每次编译都显式写标准版本。
- 每次都打开警告。
- 看到警告时不要下意识忽略。
- 尽量让每个示例都能独立编译运行。
- 先理解错误信息属于哪个阶段，再决定如何修改。

## 常见问题

### 为什么代码在某台机器能编译，在另一台不行

常见原因包括：

- 编译器版本不同
- 默认标准不同
- 包含目录配置不同
- 平台头文件或系统库差异

### 为什么明明 `#include` 了头文件还是报错

因为头文件只负责把声明带进来，不保证实现已经被正确链接。

### 为什么我只改了一行头文件，很多文件都重新编译了

因为头文件常常会被多个源文件包含，任何改动都可能影响一大片翻译单元。

## 小结补充

环境准备真正要建立的，不只是“我装了编译器”，而是：

- 我知道自己在用哪套工具链
- 我知道如何显式启用 `C++11`
- 我知道编译错误和链接错误的区别
- 我知道如何打开警告、调试信息和 sanitizer
- 我知道最小多文件程序是怎样被构建出来的

这套基础一旦立住，后面学习语言细节时会稳定很多。
